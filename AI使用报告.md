# AI 使用报告

## 1. 使用的 AI 工具及模型

本项目使用 AI 作为研发辅助工具，用于方案整理、代码实现、问题定位、测试补充和交付文档整理。

| 工具 | 模型 / 能力 | 用途 |
| --- | --- | --- |
| Cursor Agent（主会话） | 多模型（随订阅与任务切换） | 架构设计、后端实现、前端开发、调试与文档编写；集成子任务结论并落地改动 |
| **Subagent / Task（子代理）** | 并行子进程任务 | 大范围探索、只读调研、Shell/测试执行；与主会话**并行**缩短墙钟时间 |
| Cursor 检索与编辑能力 | IDE 内置能力 | 跨文件搜索、重构、回归修复 |
| **Agent Skills** | 项目与工程规范固化（`SKILL.md`） | **规范常规需求**：账本/风控/事务边界等不可协商规则一键注入上下文 |

---

## 2. Subagent 多代理并行与 Skills 规范

本节说明在本项目中如何**并行使用子代理**加速研发，以及如何用 **Skills** 把重复性约束从「每次手写长 prompt」变成可复用规范。

### 2.1 Subagent：多代理并行开发

- **定位**：在 Cursor 等环境中，将**探索、检索、跑命令、分模块阅读**拆成独立子任务（Subagent / Task），与主会话**同时推进**，避免单会话上下文被海量文件列表占满。  
- **典型并行切分**（示例）：  
  - 子任务 A：只读梳理 `backend/internal/domain/liquidation` 与 Outbox 事件消费关系；  
  - 子任务 B：对照 `spec/api/openapi.yaml` 核对 `trading_handler` 与前端 `shared/api.ts`；  
  - 子任务 C：在沙箱中执行 `go test ./...` 或定向包测试并带回失败摘要；  
  - **主会话**：合并结论、修改代码、跑 `trading-regression-local.sh` 或 E2E。  
- **收益**：墙钟时间显著缩短；探索类信息与实现类指令分离，减少「读完半个仓库再改三行」的浪费。  
- **边界**：**同一文件的并发写入**仍应避免；高风险资金逻辑以主会话或单线串行修改 + 人工审阅为主，子代理更适合**只读审查、搜索、跑测、出差异报告**。

### 2.2 Skills：规范「常规需求」的默认约束

- **是什么**：Agent Skills 以 `SKILL.md` 等形式描述**何时启用、必须遵守的不变量与检查清单**；模型在相关任务上自动或按 `@` 引用加载，相当于可版本化的「项目副脑」。  
- **在本项目中的用法**：  
  - **`rgperp-ledger-guardrails`**：凡触及充值、提现、订单保证金、清算、资金费、账本分录、Outbox 与快照等，优先挂载；强调**账本为唯一资金真相**、`ledger_tx` / `ledger_entries` 守恒、关键路径同事务、幂等与补偿分录等。  
  - **`ai-engineering-guardrails`**：通用工程护栏（正确性、失败恢复、生产就绪度），用于设计评审、重构与跨栈任务。  
  - 与 **Cursor Rules / 用户规则** 叠加：Rules 管会话级行为（如必须自跑命令、少扩散改动面），Skills 管**领域与资金安全**的深度约束。  
- **对「常规需求」的价值**：例如「补一个 Handler」「加表驱动单测」「审查本次改动是否绕过账本」——不必每次粘贴大段红线，**先声明 Skill 再写具体任务**，输出更一致、漏检更少。  
- **维护建议**：Skill 与 `spec/`、`docs/技术架构-对外版.md` 冲突时，以规格与实现为准，及时修订 Skill 文本，避免规范漂移。

### 2.3 主会话、Subagent、Skills 如何配合

| 角色 | 职责 |
| --- | --- |
| **Skills** | 任务开始前锁定「不能做什么、必须先想什么」（尤其资金与一致性）。 |
| **Subagent** | 并行完成取证、搜索、只读审查草案、长命令；返回结构化摘要供主会话采纳。 |
| **主会话** | 定方案、改代码、解决合并冲突、触发完整回归；对子代理结论做**交叉验证**。 |

### 2.4 典型 Subagent 委托 prompt 示例

```text
【子任务 · 只读探索】在 RGPerp 仓库内梳理 liquidator-worker 从 Outbox 消费到写入 liquidations /
liquidation_items 的调用链，列出涉及的文件路径、主要函数与事件类型。禁止修改任何文件；
输出结构化 Markdown，便于主会话直接引用。
```

```text
【子任务 · 并行执行】在 backend 下执行 go test ./internal/domain/... -count=1（或指定包路径），
将失败用例的完整输出摘要带回，并给出不超过 5 条的可能根因假设。不要在本子任务中改业务代码。
```

```text
【子任务 · 对照契约】对照 spec/api/openapi.yaml 与 frontend/src/shared/api.ts，列出交易相关字段
命名或类型不一致之处，仅输出差异表，由主会话决定如何修复。
```

### 2.5 结合 Skills 的常规需求 prompt 示例

```text
【已挂载 rgperp-ledger-guardrails】本次改动涉及提现冻结与解冻。请先按 Skill 中的账本不变量自检：
借贷是否平衡、幂等键、是否与其他资产同事务；再给出最小 diff 实现与建议单测场景。
```

```text
【已挂载 ai-engineering-guardrails】请对以下 Outbox 消费函数做失败恢复与幂等审查，输出按严重度排序的问题清单，
并标注每条建议对应 Skill 中的哪类护栏（可观测性 / 重试 / 数据一致性等）。
```

## 3. 关键 prompts 示例

以下 prompts 代表 AI 在 **RGPerp** 仓库中承担的主要工作类型。实际使用时，这些 prompts 通常会附带题目原文、技术栈约束、当前代码上下文和阶段性目标。**涉及账本/清算/订单写路径时，建议与 §2.2 中的 Skills 同用。**

### 3.1 需求拆解与产品结构设计

```text
下面是完整题目要求与技术约束。请你以资深产品经理和系统架构师的身份完成需求拆解。

目标：
1. 提炼核心功能模块
2. 输出完整业务流程
3. 标出高风险环节与待确认问题
4. 给出适合当前交付目标的分阶段实施路线

要求：
- 输出必须覆盖前端、后端、链上 Vault、Indexer、行情采集、订单写路径与执行 Worker、风控 Worker、清算 Worker、资金费 Worker、对冲 Worker、统一账本与 MySQL / Redis
- 如果需求有模糊点，先列出必须确认的问题
- 如果可以做多种方案，请给出推荐方案，并说明推荐理由
- 口吻采用正式技术文档风格，不复述题目原文，不使用讨论式措辞

输出格式：
1. 核心功能结构图
2. 业务流程
3. 风险点与关键设计决策
4. 分阶段实施建议
```

### 3.2 总体架构与模块职责定义（RGPerp）

```text
请基于以下技术栈与约束，归纳当前 RGPerp 仓库中「链上托管、链下 CFD 交易、链下风控与清算、外部行情与可选对冲」的系统架构。

约束（与仓库一致）：
- 前端：React + Vite + TypeScript + Ant Design
- 后端：Go + Gin + GORM；模块化单体 api-server + 多进程 Worker
- 数据层：MySQL + Redis；
- 合约：Solidity + Foundry
- 异步编排：事务性 Outbox（outbox_events + message_consumptions）

要求：
- 明确 Web、api-server、Vault/路由合约、indexer、market-data、order-executor、risk-engine、liquidator、funding、hedger、migrator 的职责边界
- 给出顶层架构图、模块拆分、数据流和关键状态流转
- 说明哪些状态在链上，哪些在链下，以及统一账本（ledger）与订单/成交/仓位的关系
- 说明同步 HTTP 与异步 Worker 的边界；Outbox 如何保证与主事务一致写入
- 输出必须适合写入 docs/技术架构-对外版.md 或 Architecture Document
```

### 3.3 交易模型选型与方案比较

```text
请比较 CFD 参考价模式与订单簿撮合模式在当前 RGPerp 目标下的差异，并给出明确推荐。

比较维度：
- 实现复杂度
- 系统稳定性
- 风控可控性
- 对冲难度
- 演示效果
- 交付风险

要求：
- 结合当前项目是「完整可运行系统」而不是单纯概念验证来分析
- 明确说明两种方案下订单、仓位、PnL、风险和对冲的实现差异
- 最后必须给出一个推荐方案，并说明为什么该方案更适合当前代码架构（api-server 单事务成交路径 + executor Worker）和交付节奏
```

### 3.4 保证金、PnL 与清算逻辑设计

```text
请设计（或对照现有实现审查）永续 CFD 场景下的保证金、已实现盈亏、未实现盈亏、维持保证金、风险率、清算与保险基金之间的关系。

要求：
- 同时覆盖 cross 和 isolated 两种保证金模式
- 给出核心公式与字段归属（账户读模型 vs 仓位）
- 给出从开仓、持仓、价格波动、风险升级到强平的完整状态流转
- 说明提现前风控应如何判断用户可提现额度
- 说明清算 Worker 与 Outbox 事件（如 risk.liquidation.triggered）的衔接
- 输出要求适合直接指导后端 domain 服务与 spec/db/mysql-ddl.sql 对齐
```

### 3.5 订单逻辑审查与一致性校验

```text
请审查当前下单与执行相关逻辑，重点检查以下环节是否一致：
- 开仓
- 同向加仓
- 反向开仓
- 部分平仓
- 全量平仓
- 反手（若产品支持）

要求：
- 不要只做代码解释，要以代码审查的方式指出潜在不一致和具体风险
- 如果发现问题，请说明会导致什么资金或仓位错误
- 结合 Idempotency-Key 与幂等语义，说明重复请求下的期望行为
- 如果需要测试，请补充单元测试与 deploy/scripts/trading-regression-local.sh 类回归建议
- 输出优先级按严重程度排序
```

### 3.6 对冲逻辑校验与异常分析

```text
请审查当前系统的对冲逻辑。系统按净敞口统一对冲意图组织 hedge 领域与 hedger-worker。

请重点检查：
- 内部净敞口计算是否正确
- 外部对冲方向是否与净敞口一致
- hedge 相关表状态机是否自洽
- 失败重试和管理端操作是否会造成重复对冲或错误状态残留
- risk 域产出的 hedge.requested 类 Outbox 与 hedger-worker 消费是否对齐

要求：
- 用具体示例说明用户净多、用户净空时平台内部风险与外部对冲方向
- 如果发现逻辑错误，请指出应改「目标定义」还是「执行细节」
- 输出必须可直接指导 hedger-worker 与管理端相关修复
```

### 3.7 清算 / 交易 runbook 与演示路径生成

```text
请基于当前 RGPerp 代码与 docker-compose 环境，生成一份可重复的测试 runbook。

目标：
- 从启动环境开始
- 覆盖登录、充值（或本地入账路径）、开仓、制造风险或使用演示数据、触发清算（如适用）、验证清算记录与账户摘要
- 说明如何配合 frontend/e2e 与 trading-regression-local.sh

要求：
- 给出可执行命令或脚本入口
- 说明每一步应观察哪些 API、数据库记录和前端变化
- 说明成功标准
- 优先覆盖最关键的风险链路，而不是罗列所有接口
```

### 3.8 交付文档统一润色

```text
请基于当前实际实现，统一整理 README、对外技术架构白皮书、OpenAPI/DDL 引用说明与 Demo 文档。

要求：
- 文档口吻采用专业、对外、肯定式表达
- 去掉不确定、讨论式、路线规划式措辞
- 所有描述必须与当前真实系统状态一致
- 保留已知限制，但必须写成明确的当前限制，而不是「以后可能实现」
- 质量保障章节须与仓库内测试、回归脚本、E2E 路径一致
- 输出应直接可用于项目交付或招投标附件
```

## 4. AI 生成代码比例估计

以下比例为 **RGPerp 项目整体经验估计**（非行级度量，仅供内部对齐与对外说明口径参考）。

- 架构与交付文档：`85% - 95%`
- 前端页面与样式调整：`85% - 100%`
- 后端 Handler、仓储样板与测试补充：`80% - 95%`
- 核心交易、风控、清算、账本与 Outbox 编排：`70% - 90%`（以 AI 生成候选实现 + 多轮人工修正为主）
- 测试用例、回归脚本与 E2E 草稿：`85% - 95%`

总体上，AI 在结构化文档、页面框架、基础样板和测试补充上贡献较高；在资金、风控、清算、账本守恒与异步一致性等高风险模块中，AI 主要提供候选实现、审查清单与用例建议，**最终正确性依赖人工复查与真实联调**。  
**Subagent** 拉高探索与跑测的并行度，**Skills** 拉高常规需求在安全与一致性上的下限；二者不改变「核心资金逻辑必须人审」的结论。

## 5. AI 帮助解决的问题

### 5.1 系统拆解与技术路线收敛

AI 用于快速整理复杂需求，将系统拆分为：

- 前端交易终端与 Explorer / 管理端
- **api-server**（同步 API 聚合）
- Vault / 充值路由等合约
- **indexer**
- **market-data**（多源行情与标记价）
- **order-executor-worker**（挂单、触发单等）
- **risk-engine-worker**
- **liquidator-worker**
- **hedger-worker**
- **funding-worker**
- **统一账本**与 **migrator**

同时用于对比关键技术路线，例如：

- CFD 参考价模式 vs 订单簿撮合
- 净敞口统一对冲 vs 逐笔对冲
- 事务性 Outbox vs 直接消息投递
- Binance / Hyperliquid 等价格源与符号映射策略

### 5.2 后端实现加速

AI 参与完成或迭代过多轮修订的模块包括但不限于：

- 钱包登录与 JWT / 签名验证
- 订单、成交、仓位与账户读模型
- 统一账本分录与钱包服务
- 行情采集、缓存与符号状态
- 资金费率批次与入账
- 对冲意图与 hedger-worker 衔接
- 风险快照重算与清算调度消费
- HTTP 层 trading / wallet / admin 等 Handler 与中间件

### 5.3 前端页面与交互实现

AI 参与的前端工作包括但不限于：

- 交易页、资产与历史、Explorer、管理端页面
- 与 `shared/api.ts`、领域类型的对接
- 图表与行情展示相关集成
- Playwright E2E（交易、清算、提现等场景）草稿与稳定化

### 5.4 问题定位与回归测试

AI 用于定位并辅助修复过以下类型问题（示例性质，随迭代增减）：

- 保证金与风险口径不一致
- 幂等与重复下单边界
- 触发单、撤单与仓位变化不同步
- Outbox 消费与业务幂等配合疏漏
- 清算或资金费链路中与账本模板不一致

对应修复通常伴随 **Go 单测**、`trading-regression-local.sh` 或 **E2E** 补充。

### 5.5 Subagent 与 Skills 带来的增量收益

- **Subagent**：并行完成「全库语义搜索、多包测试、契约对照」等耗时活动，主会话专注于合并结论与落码，整体迭代节奏更顺。  
- **Skills**：使「补 Handler / 加单测 / 审查是否绕过账本」等**高频、重复形态**的需求输出稳定，减少同一 invariant 在多轮对话中被遗漏的概率。  
- **组合**：例如清算问题定位时，一子代理追溯 Outbox 与领域调用链，另一子代理跑定向测试；主会话结合 **ledger guardrails** 审分录后再改代码。

## 6. AI 无法独立解决的问题

### 6.1 资金安全与风控边界

保证金、清算、对冲、账本分录与提现属于高风险业务逻辑。AI 可以辅助推导公式和补样板代码，但不能替代人工对以下问题的最终判断：

- 风险阈值与罚金参数是否合理
- 清算条件与保险基金扣补是否闭环
- 账务模板是否在所有场景下守恒
- 外部对冲或链上失败后的系统状态是否仍可控

### 6.2 真实联调与环境验证

AI 不能替代：

- 合约部署与本地链联调
- Docker Compose 全栈验证
- `trading-regression-local.sh` 在真实环境中的通过性
- 强平、提现、资金费率等端到端联调
- 压测工具（如 api-stress）下的延迟与容量结论

### 6.3 最终产品判断

前端视觉层级、交互细节与文案风格由多轮人工反馈驱动。AI 用于加速实现与草稿，但最终呈现与对外口径由人工统一确定。

### 6.4 Subagent 与 Skills 的局限（仍需人工兜底）

- **Subagent**：结论来自子任务局部上下文，可能与当前分支或环境不一致；**不得未经交叉验证即作为架构或资金逻辑的最终依据**。  
- **Skills**：为版本化文本，若与 `spec/`、实现或监管要求不同步，会产生过严或过松的指示；**以仓库规格与代码为准，Skill 需随基线迭代修订**。

### 6.5 模型与产品形态的固有局限（上下文 · 能力波动 · 时延）

以下问题来自**当前大模型与 IDE 助手产品的普遍机制**，无法通过「换一个好 prompt」彻底消除，只能在流程上缓解（拆任务、子代理、规格锚定、自动化测试）。

| 局限 | 表现与影响 | 缓解思路（工程侧） |
| --- | --- | --- |
| **上下文压缩 / 窗口截断** | 长会话中较早的约束、文件版本或已否决方案可能被摘要或丢弃，导致后续回答与既定决策矛盾，或在大型 PR 上漏看关键 diff。 | 关键约束写入 **Skills / Rules / 提交说明**；大改动拆 PR；重要结论用仓库内文档或注释固定；长线程新开会话并显式粘贴基线。 |
| **「降智」与能力波动** | 负载路由、模型版本或策略调整时，同类任务可能出现推理变浅、跳过边界情况、过度「想当然」补全；不同轮次质量不一致。 | **高风险路径不依赖单次模型输出**；强制单测/回归；代码审查；对账本与 Outbox 等用 checklist 二次过一遍。 |
| **输出慢与交互时延** | 长上下文、多文件编辑或复杂 Agent 回合下，首 token 与整段生成变慢，打断心流；并行子任务仍受配额与队列影响。 | 用 **Subagent** 把可并行的只读/跑测拆出去；控制单次请求范围；避免在助手内展开超大无关文件。 |
| **幻觉与过时知识** | 可能编造不存在的 API、包名或链上细节；训练数据滞后于当前依赖版本。 | **以仓库与官方文档为准**；生成后立刻 `go test` / 编译；对外契约以 `spec/` 为单一真相源。 |
| **非确定性** | 同 prompt 多轮输出可能不同，不利于「可复现的审查结论」。 | 将必须通过项写进自动化测试与 CI；审查结论关键处人工落笔或记 issue。 |

---

## 7. 人工校正方式

本项目采用以下方式保证 AI 参与下的交付质量：

1. 资金、风控、清算、账本与 Outbox 类逻辑全部经过人工复查；**Skills 中的护栏是必要非充分条件**，不能替代人审。
2. 重要修复补充单元测试、API 回归脚本或 E2E；**Subagent 跑测结果**须由主会话或人工复核后再合并。
3. 关键链路进行真实联调，包括：登录、充值/入账、下单、成交与仓位、资金费、清算（如启用）、提现（如启用）、Explorer 事件一致性。
4. 对长会话或超大 diff：主动**新开上下文**并重申关键约束，避免仅依赖模型对早期轮次的记忆（见 **§6.5**）。

---

## 8. 总结

AI 在 RGPerp 中的价值不仅在于单会话生成代码与文档，还在于 **Subagent 多路并行**（探索、跑测、契约对照与主会话落码解耦）与 **Skills 对常规需求的规范化**（尤其资金与账本不变量），在相同墙钟时间内完成更多验证步骤、降低重复性遗漏。与此同时，**上下文压缩、能力波动与时延**等产品层局限需要通过拆任务、规格锚定与自动化测试来管理，而非假设模型「始终记得全部细节」。交付质量仍取决于 **人工设计判断、护栏与实现的一致性维护、以及真实联调**。
