**文档基线**：原始文档《技术设计文档.md》  
**版本**：v1.0（扩充版）

> 本版文档在原始需求基础上完成三件事：
> 1. 把模糊描述收敛为可实施规格；
> 2. 增补生产级 CFD 永续合约交易所必须具备但原文未写明的能力；

## 1. 文档目标与范围

### 1.1 文档目标

本系统是一个**基于 CFD（差价合约）模型**的永续合约交易所，采用：

- **USDC 单一保证金结算**；
- **EVM 地址登录**；
- **链上充值 / 提现 + 链下统一账本**；
- **外部行情聚合定价 + 平台内部撮合/成交计算**；
- **资金费率、清算、统一账本、审计追踪、运营看板**；
- **支持生产交付、同时支持 docker compose 一键拉起的 review 环境**。

本文档的定位不是产品 PRD，也不是仅供讨论的高层架构稿，而是**可直接指导实现的系统需求与技术设计规格**。

### 1.2 核心设计原则

1. **资金安全优先于成交体验**。
2. **账本优先于余额缓存**；任何资产变化都必须有可追踪账务记录。
3. **风控优先于交易**；宁可拒单、限流、降级，也不能产生坏账扩散。
4. **标记价格优先于成交价格**；清算、TP/SL、风险评估必须基于抗操纵价格。
5. **强一致写链路，最终一致读链路**；交易、账本、仓位必须同步一致，报表、看板、消息可异步。
6. **可回放、可审计、可对账**；任何资金变更、配置变更、清算事件、管理员动作必须可追溯。
7. **review 模式与 production 模式严格隔离**；测试便利性不能污染生产安全边界。

### 1.3 范围内能力

本版文档覆盖以下能力：

- 账户、认证、权限；
- 统一账本、资金子账户、幂等、补账、对账；
- USDC 多链充值、提现、归集、热温冷钱包；
- CFD 订单、成交、限价/市价/触发单、仓位、PnL、保证金；
- 行情聚合、指数价、标记价、成交价；
- 资金费率、风控、强平、保险基金、ADL；
- 平台净敞口与外部对冲；
- Explorer、报表、后台、配置中心；
- 高可用、性能、安全、监控、审计、发布与 docker compose review 环境。

### 1.4 范围外能力

以下内容不在本期强制范围内，但可预留扩展位：

- 现货交易；
- 期权、交割合约；
- 法币出入金；
- 理财、借贷、跟单、机器人市场；
- 推荐返佣、VIP 复杂营销体系；
- 多币种保证金组合保证金。

## 2. 原始文档中的模糊项与本版澄清

### 2.1 “覆盖所有在两平台均存在的合约交易对”

- **原问题**：两平台未明确，交易对映射、下线逻辑、价格源优先级也未定义。
- **本版决议**：默认两平台为 **Binance 与 Hyperliquid**；但价格源集合必须配置化，不允许写死在业务代码中。交易对必须维护独立的 `symbol_mapping`，用于处理不同平台的命名差异、精度差异、最小单位差异、交易时段差异和下线差异。

### 2.2 “所有操作必须生成唯一标识（txHash）”

- **原问题**：充值/提现有链上 `txHash`，但下单、成交、转账、资金费、清算是链下行为，不应强行复用链上哈希概念。
- **本版决议**：统一拆分为：
  - `event_id`：系统内全局唯一业务事件 ID；
  - `ledger_tx_id`：账本事务 ID；
  - `chain_tx_hash`：仅链上动作存在；
  - `order_id / fill_id / position_id / liquidation_id / funding_batch_id`：领域对象 ID；
  - `internal_seq`：全局递增序列，用于排序与回放。

### 2.3 “按区块查询”

- **原问题**：链下交易不存在链上区块高度。
- **本版决议**：Explorer 同时支持两类查询：
  - 链上事件按 `chain_id + block_height + tx_hash` 查询；
  - 链下事件按 `internal_seq / event_id / time_range / address` 查询。

### 2.4 “账户只有余额、占用保证金、冻结资金”

- **原问题**：生产系统还需要区分订单冻结、持仓保证金、提现冻结、待结算费用、补偿流水等。
- **本版决议**：对外展示仍可收敛为“余额 / 占用保证金 / 冻结资金”，但内部必须拆为多个资金子账户，至少包括：
  - `USER_WALLET`（可用资金）；
  - `USER_ORDER_MARGIN`（订单冻结）；
  - `USER_POSITION_MARGIN`（持仓占用）；
  - `USER_WITHDRAW_HOLD`（提现冻结）；
  - 必要时可扩展 `USER_BONUS / USER_ADJUSTMENT_HOLD`。

### 2.5 “开仓/平仓/强平流程和开关仓保持一致”

- **原问题**：缺少在 CFD 模式下的订单校验、限价单触发规则、清算执行路径、部分强平、坏账处理。
- **本版决议**：
  - 普通开平仓走**订单执行引擎**；
  - 强平走**清算执行器**，具有更高优先级与专用风控；
  - 清算前必须取消挂单、释放订单冻结；
  - 支持分阶段强平、保险基金兜底、ADL 最后兜底。

### 2.6 “Funding Rate 取 Binance 和 Hyperliquid 的平均值”

- **原问题**：不同平台资金费率结算周期可能不同；部分品种不一定双边都有数据。
- **本版决议**：
  - 资金费率源必须先**归一到统一周期（默认 1 小时）**；
  - 同时存在时做等权平均，权重可配置；
  - 仅单源可用时走单源兜底；
  - 无有效源时进入降级模式：暂停开新仓或切换到内部参考模型。

### 2.7 “支持 Commodities / Equities”

- **原问题**：商品和股票 CFD 涉及交易时段、节假日、停牌、分红、拆股、合约展期等问题，原文未定义。
- **本版决议**：品种必须带有 `asset_class` 与 `session_policy`：
  - `ALWAYS_OPEN`：如大多数 Crypto；
  - `REFERENCE_MARKET_HOURS`：仅参考市场开盘时允许开仓；
  - `SYNTHETIC_EXTENDED_HOURS`：闭市允许有限报价，但扩大点差、降低杠杆、可强制 reduce-only。
  - Equities 必须支持**分红、拆股、合股、停牌、退市**处理；Commodities 必须支持**基准合约切换/展期**策略。

### 2.8 “系统中的全部资金 = 链上资产”

- **原问题**：若平台做外部对冲或将资金打到上游交易所/经纪商，真实资产不只存在链上地址。
- **本版决议**：生产口径应改为：

```text
用户负债 + 平台权益 = 链上托管资产 + 上游对冲账户资产 + 在途资产
```

系统必须对**链上钱包、外部对冲账户、在途提现、待归集余额**分别建账和对账。

### 2.9 “提供接口给指定地址增加测试资金（可不鉴权）”

- **原问题**：这一要求仅适用于 review/dev，不得进入生产。
- **本版决议**：
  - 仅 `review` / `dev` profile 启用测试水龙头；
  - 生产环境编译与运行参数必须强制关闭；
  - 水龙头也必须写入账本，且有地址/时间窗口限额。

### 2.10 “高可用、可扩展、可恢复”

- **原问题**：缺少可量化目标。
- **本版决议**：必须定义 SLO / RTO / RPO / 性能指标 / 降级策略 / 监控指标 / 演练要求，见第 17 节。

### 2.11 “所有余额修改与账本变化必须原子性”

- **原问题**：未区分源账本与余额缓存。
- **本版决议**：
  - **账本 + 仓位 + 订单状态**属于强一致写链路；
  - 用户“余额快照 / 风险看板 / Explorer 索引 / 报表”可由事件流异步构建；
  - 余额缓存只能是读优化，不能替代账本。

### 2.12 “全仓模式 + 多空双向持仓”

- **原问题**：未定义仓位主键与风险计算方式。
- **本版决议**：
  - 风险模式默认 **Cross Margin**；
  - 仓位模式支持 **Hedge Mode**；
  - 仓位主键为 `user_id + symbol + side`；
  - 多空仓位分别维护 `qty / avg_entry / realized_pnl / funding_accrual / liq_price`；
  - 账户权益在 Cross 维度统一计算。

## 3. 总体系统架构

### 3.1 架构分层

系统按职责划分为以下域：

1. **接入层**：Web / App API、WebSocket、管理后台 API、Explorer API。
2. **身份与权限层**：EVM 登录、会话、角色权限、设备风险、管理后台二次认证。
3. **交易核心层**：订单管理、执行引擎、仓位服务、风险引擎、清算服务、资金费率服务。
4. **资金核心层**：账本服务、充值监听、提现处理、归集、对账、热温冷钱包、测试水龙头。
5. **行情与价格层**：多源行情接入、标准化、指数价、标记价、成交价、异常熔断。
6. **平台风险层**：净敞口、对冲路由、上游连接器、保险基金、ADL。
7. **数据与审计层**：Explorer、报表、数据看板、审计日志、回放、归档。
8. **基础设施层**：数据库、缓存、消息总线、对象存储、监控告警、配置中心、密钥管理。

### 3.2 一致性边界

必须明确两条链路：

**强一致同步链路**（必须事务成功后才能返回成功）：

- 登录签名验证后的会话创建；
- 下单风控校验；
- 成交后的订单状态更新；
- 持仓更新；
- 账本入账；
- 提现冻结；
- 清算执行；
- 管理员人工补账。

**最终一致异步链路**（允许秒级延迟）：

- Explorer 索引构建；
- 数据看板聚合；
- 风险报表；
- 用户通知；
- 审计归档；
- 搜索索引；
- 对账快照。

### 3.3 推荐服务拆分

| 服务 | 核心职责 |
| --- | --- |
| Gateway/API | REST、WS、鉴权、限流、签名校验入口 |
| Auth Service | Nonce、签名校验、会话、设备/IP 风险 |
| Account Service | 用户资料、状态、账户视图、权限联动 |
| Ledger Service | 账本事务、资金子账户、余额快照、补账、幂等 |
| Wallet Service | 充值监听、提现、归集、钱包管理、链上对账 |
| Market Data Service | 行情接入、清洗、聚合、指数价/标记价 |
| Order Service | 下单、改单、撤单、触发单管理 |
| Execution Engine | CFD 成交价计算、成交生成、限价单触发 |
| Position Service | 持仓、PnL、保证金、仓位视图 |
| Risk Engine | 预交易校验、账户风险率、限仓、限杠杆 |
| Liquidation Engine | 订单撤销、部分强平、全平、罚金、坏账路径 |
| Funding Service | 资金费率采集、归一、结算、历史存档 |
| Exposure/Hedge Service | 平台净敞口、对冲、上游账户监控 |
| Explorer Service | 事件搜索、流水展示、链上链下关联 |
| Admin/Config Service | 参数动态配置、审批、回滚、后台操作 |
| Notification Service | 订单成交、风险预警、入金/出金通知 |
| Reconciliation Service | 钱包/账本/上游对冲账户三方对账 |

### 3.4 数据存储建议（推荐实现）

为满足生产交付与本地 review 双重需求，建议采用：

- **PostgreSQL**：订单、仓位、账本、资金流水、配置、审计元数据；（可降级为mysql存储）
- **Redis**：会话、nonce、热点价格缓存、限流、幂等短缓存；
- **Kafka / Redpanda**：异步事件总线；（可降级为rabbit mq）
- **ClickHouse**：Explorer 聚合、风控报表、运营看板；（可降级为mysql存储）
- **对象存储**：审计归档、导出文件、对账快照；（可降级为mysql存储）
> 单机 review 环境可以裁剪为 PostgreSQL + Redis + 单节点消息总线；但接口与数据结构必须与生产保持一致，不允许为了 demo 改业务语义。

## 4. 账户、身份与权限系统

### 4.1 账户结构

对外：

- 一个 EVM 地址对应一个平台主账户；
- 当前版本不开放用户自定义子账户；
- 支持账户之间内部转账。

对内：

- 同一用户账户会拆分为多个**账务子账户**（钱包、订单冻结、持仓保证金、提现冻结等）；
- 这些子账户仅用于内部记账，对外统一聚合展示。

### 4.2 登录与认证要求

必须采用 **EVM 地址签名登录**，建议使用“nonce 挑战 + 标准化签名消息”的方式。

**登录流程**：

1. 用户请求登录 nonce；
2. 服务端返回一次性 nonce、域名、链 ID、过期时间；
3. 用户钱包签名；
4. 服务端校验签名、nonce、域名绑定、过期时间、防重放；
5. 创建会话并返回 access token + refresh token；
6. 记录设备指纹、IP、UA、最近登录时间。

**强制要求**：

- nonce 只能使用一次；
- 登录消息必须绑定当前域名/环境，防止跨站复用；
- access token 短时有效，refresh token 可撤销；
- 提现、后台高危操作应要求二次确认（可为再次签名、TOTP、WebAuthn、邮箱验证码中的一种或组合）；
- 管理后台必须与用户前台权限域隔离。

### 4.3 账户状态机

账户至少包含以下状态：

- `ACTIVE`：正常可交易、可出入金；
- `LOGIN_RESTRICTED`：限制登录或要求二次验证；
- `TRADE_ONLY`：允许交易，不允许提现；
- `WITHDRAW_ONLY`：限制新开仓，仅允许平仓/提现；
- `FROZEN`：完全冻结；
- `CLOSED`：逻辑关闭，不可再登录。

### 4.4 权限模型

必须至少支持以下角色：

- `USER`：普通用户；
- `SUPPORT_READONLY`：客服只读；
- `OPS`：运营；
- `RISK_ADMIN`：风控参数与风险处置；
- `FINANCE_ADMIN`：提现审核、资金对账；
- `SYSTEM_ADMIN`：系统级配置；
- `AUDITOR`：审计只读；
- `SECURITY_ADMIN`：安全应急、会话封禁、密钥轮换联动。

高危权限必须满足：

- 最小权限原则；
- 审批链；
- 操作审计；
- 双人复核（提现总闸、人工补账、符号下线、风险参数调整建议强制双签）。

## 5. 统一账本与资金系统

### 5.1 账本设计原则

1. **所有资金变化必须写账本**；
2. **禁止直接 UPDATE 余额作为资金真相**；
3. **每个账本事务必须满足按资产维度守恒**；
4. **任何修正必须用冲正/补偿分录完成**；
5. **账本事务与业务对象变更必须原子提交**；
6. **账本是 source of truth，余额快照只是读优化**。

### 5.2 账本事务守恒规则

对于同一个 `ledger_tx_id + asset`，必须满足：

```text
Σ ledger_entries.amount = 0
```

注意这里的守恒是**按资产维度**守恒，不能把不同资产相互抵消。当前一期虽然只支持 USDC，但系统层面必须预留多资产扩展能力。

### 5.3 内部账务子账户

#### 用户侧账务账户

| 账户 | 说明 |
| --- | --- |
| USER_WALLET | 用户可用资金 |
| USER_ORDER_MARGIN | 挂单冻结资金 |
| USER_POSITION_MARGIN | 已成交持仓占用资金 |
| USER_WITHDRAW_HOLD | 提现审核/广播中的冻结资金 |
| USER_BONUS（可选） | 活动赠金或补贴资金 |

#### 平台侧账务账户

| 账户 | 说明 |
| --- | --- |
| SYSTEM_POOL | 平台作为 CFD 对手方的盈亏池 |
| FUNDING_POOL | 资金费暂存池，理论上结算后趋近于 0 |
| TRADING_FEE_ACCOUNT | 手续费收入 |
| WITHDRAW_FEE_ACCOUNT | 提现手续费收入 |
| PENALTY_ACCOUNT | 强平罚金收入 |
| INSURANCE_FUND | 保险基金，用于兜底坏账 |
| HEDGE_COLLATERAL_ACCOUNT | 外部对冲账户抵押资产镜像 |
| HEDGE_PNL_ACCOUNT | 外部对冲已实现盈亏镜像 |
| ROUNDING_DIFF_ACCOUNT | 极小数舍入差异专用账户（需严格限额） |
| TEST_FAUCET_POOL | review/dev 环境测试资金池 |

#### 托管与链路镜像账户

| 账户 | 说明 |
| --- | --- |
| CUSTODY_HOT | 热钱包镜像 |
| CUSTODY_WARM | 温钱包镜像 |
| CUSTODY_COLD | 冷钱包镜像 |
| SWEEP_IN_TRANSIT | 归集中在途 |
| WITHDRAW_IN_TRANSIT | 提现链路在途 |
| DEPOSIT_PENDING_CONFIRM | 充值确认中 |

> 说明：原始文档中的 `SYSTEM_POOL` 不能同时承担“用户资产镜像、对手方盈亏池、真实钱包资产镜像”三种职责，否则账会混乱。本版明确拆开。

### 5.4 账本主表字段

每一条账本分录至少包含：

- `ledger_entry_id`
- `ledger_tx_id`
- `event_id`
- `account_id`
- `user_id`（若适用）
- `asset`
- `amount`
- `biz_type`（DEPOSIT / WITHDRAW / TRANSFER / ORDER_HOLD / OPEN / CLOSE / FUNDING / LIQUIDATION / FEE / ADJUSTMENT ...）
- `biz_ref_id`（订单号、仓位号、链上哈希等）
- `idempotency_key`
- `trace_id`
- `operator`（system / user / admin / job）
- `created_at`
- `metadata_json`

建议配套以下结构：

- `ledger_tx`：账本事务头；
- `ledger_entry`：账本分录明细；
- `account_balance_snapshot`：余额快照（仅缓存）；
- `ledger_outbox`：事务消息外盒。

### 5.5 余额与权益口径

统一口径如下：

```text
账户抵押余额 = USER_WALLET + USER_ORDER_MARGIN + USER_POSITION_MARGIN + USER_WITHDRAW_HOLD + USER_BONUS
账户权益 = 账户抵押余额 + Σ未实现PNL - Σ待扣费用 - Σ待扣资金费
可交易余额 = 账户权益 - 当前持仓初始保证金需求 - 挂单保证金冻结 - 提现冻结 - 风险缓冲
```

说明：

- `账户抵押余额` 是账本可回溯口径；
- `账户权益` 是风险引擎口径；
- `可交易余额` 是下单校验口径；
- `liquidation_price` 是派生值，不作为会计真相。

### 5.6 舍入与精度要求

生产级实现必须满足：

- 禁止使用二进制浮点做资金与价格计算；
- 推荐数据库使用 `DECIMAL(38, 18)` 或整型最小单位方案；
- USDC 展示精度为 6 位，但内部计算精度建议至少 18 位；
- 所有费率、资金费率、滑点参数必须使用高精度定点数；
- 舍入模式必须配置化，并在费率/价格/数量/保证金四类字段上分别定义；
- 舍入差异只能进入 `ROUNDING_DIFF_ACCOUNT`，并设置告警阈值。

### 5.7 典型资金流（扩充版）

#### 5.7.1 充值入账

```text
USER_WALLET           +100
DEPOSIT_PENDING_CONFIRM  -100   （确认完成前）
```

确认完成后：

```text
DEPOSIT_PENDING_CONFIRM +100
CUSTODY_HOT            -100
```

> 说明：若实现上希望一步到位入账，也必须保留 `deposit` 状态机，不允许把“检测到链上 transfer”与“最终可用资金”混为一体。

#### 5.7.2 下单冻结

```text
USER_WALLET         -100
USER_ORDER_MARGIN   +100
```

#### 5.7.3 取消挂单解冻

```text
USER_ORDER_MARGIN   -100
USER_WALLET         +100
```

#### 5.7.4 开仓成交（手续费 0.1）

```text
USER_ORDER_MARGIN     -100
USER_POSITION_MARGIN  +99.9
TRADING_FEE_ACCOUNT   +0.1
```

#### 5.7.5 平仓盈利 +20（手续费 0.1）

```text
USER_POSITION_MARGIN  -99.9
USER_WALLET          +119.8
SYSTEM_POOL          -20
TRADING_FEE_ACCOUNT  +0.1
```

#### 5.7.6 平仓亏损 -20（手续费 0.1）

```text
USER_POSITION_MARGIN  -99.9
USER_WALLET           +79.8
SYSTEM_POOL           +20
TRADING_FEE_ACCOUNT   +0.1
```

#### 5.7.7 提现申请（冻结）

```text
USER_WALLET         -100
USER_WITHDRAW_HOLD  +100
```

广播成功并扣除手续费 1：

```text
USER_WITHDRAW_HOLD      -100
WITHDRAW_IN_TRANSIT      +99
WITHDRAW_FEE_ACCOUNT     +1
```

链上确认完成：

```text
WITHDRAW_IN_TRANSIT  -99
CUSTODY_HOT          +99
```

#### 5.7.8 资金费结算

长仓支付、短仓收取示例：

```text
(Long)  USER_POSITION_MARGIN  -20
FUNDING_POOL                  +20
(Short) USER_POSITION_MARGIN  +10
FUNDING_POOL                  -10
```

周期结束后 `FUNDING_POOL` 剩余若不为 0，只能是舍入差异或异常，必须告警并在同批次内解释。

#### 5.7.9 强平（含残余退回）

```text
USER_POSITION_MARGIN   -100
SYSTEM_POOL            +94
PENALTY_ACCOUNT        +1
USER_WALLET            +5
```

若平仓后用户仍为负权益：

```text
INSURANCE_FUND         -bad_debt
SYSTEM_POOL            +bad_debt
```

保险基金不足时触发 ADL。

### 5.8 幂等、补账与回滚要求

- 每个外部请求必须携带 `idempotency_key`；
- 链上充值去重键建议为 `chain_id + tx_hash + log_index`；
- 下单去重键建议为 `user_id + client_order_id`；
- 资金费批次去重键建议为 `funding_batch_id + user_id + symbol + side`；
- 补账只能走 `ADJUSTMENT` 事务，且必须双人审批、填写原因、保留附件；
- 禁止删除账本分录；
- 禁止通过脚本直接改余额快照修账。

## 8. 行情与定价系统

### 8.1 行情源要求

- 默认主源：Binance、Hyperliquid；
- 可扩展备源；
-  REST 回补；
- 必须保存源头时间戳与本地接收时间戳；
- 必须有源级健康检查和心跳机制。

### 8.2 标准化要求

在进入定价引擎前，必须完成：

- symbol 映射；
- 小数精度标准化；
- 数量单位标准化；
- bid/ask/last/mid 统一格式；
- 源头时区与时间格式统一；
- 失效数据过滤。

### 8.3 三类核心价格

#### 8.3.1 指数价格（Index Price）

用于表达底层参考公允价格，建议采用：

```text
Index Price = valid source mid prices 的加权中位数 / 截尾均值
```

要求：

- 至少有最小有效源数量（quorum）；
- 偏离中位数超过阈值的源要被剔除；
- 指数价不可直接使用单个成交价。

#### 8.3.2 标记价格（Mark Price）

用于：

- 未实现 PnL；
- 清算；
- TP/SL 触发；
- 风险率计算。

建议：

```text
Mark Price = clamp(Index Price + EMA(Synthetic Mid - Index Price), 上下偏离上限)
```

要求：

- 必须抗操纵；
- 不能直接用最新成交价；
- 价格更新要可追溯。

#### 8.3.3 成交价格（Execution Price）

用于最终成交结算，基于：

- 外部聚合深度；
- 用户下单方向和下单量；
- 平台点差 / 滑点参数；
- 平台当前净敞口调节因子；
- 用户设置的价格保护。

示例：

```text
Buy Execution Price  = impact_ask(size) * (1 + base_spread + exposure_adj)
Sell Execution Price = impact_bid(size) * (1 - base_spread - exposure_adj)
```

### 8.4 CFD 模式下的限价单语义（原文未明确，本版补充）

由于本平台不是用户之间互相撮合，而是**平台根据外部参考深度生成成交**，因此限价单语义必须定义为：

- 用户限价单进入“待触发/待成交”池；
- 当外部合成最优买/卖价满足用户价格条件时，订单变为可成交；
- 成交价仍由执行引擎计算，但不得比用户限价更差；
- 不存在用户间撮合优先队列时，平台可采用“时间优先 + 风险优先”执行规则。

### 8.5 行情降级与熔断

必须支持：

- 单源失效自动降级；
- 多源分歧过大时暂停开新仓；
- 标记价冻结与 reduce-only 模式；
- 价格跳变超阈值时触发 symbol 熔断；
- 行情延迟超过阈值时拒绝新开仓、允许撤单和平仓。

### 8.6 行情数据落地与回放

- 必须保存关键价格快照与异常事件；
- 至少保留成交相关的报价快照以便事后审计；
- 支持按 `symbol + time_range` 回放价格序列；
- 清算触发前后的价格快照必须可查询。

## 10. 仓位、保证金与 PnL

### 10.1 仓位模型

每个仓位至少包含：

- `position_id`
- `user_id`
- `symbol`
- `side`（LONG / SHORT）
- `qty`
- `avg_entry_price`
- `mark_price`
- `notional`
- `initial_margin`
- `maintenance_margin`
- `realized_pnl`
- `unrealized_pnl`
- `funding_accrual`
- `liquidation_price`
- `bankruptcy_price`
- `status`

### 10.2 风险模式与仓位模式

- **风险模式**：本期强制支持 `Cross Margin`；
- **仓位模式**：强制支持 `Hedge Mode`；
- 同一用户在同一 symbol 可同时持有 LONG 与 SHORT 两个仓位；
- 风险计算以账户总权益为准，仓位分别展示。

### 10.3 关键公式

```text
持仓名义价值 = abs(qty) × mark_price × contract_multiplier
未实现PNL = direction_sign × qty × (mark_price - avg_entry_price) × contract_multiplier
账户权益 = 账户抵押余额 + Σ未实现PNL
风险率 = 账户权益 / (Σ维持保证金要求 + 清算费用缓冲)
```

其中：

- `direction_sign`：多仓为 +1，空仓为 -1；
- `风险率 <= 1` 进入清算；
- `liquidation_price` 与 `bankruptcy_price` 必须实时更新，但属于派生字段。

### 10.4 风险档位（Tier）

每个 symbol 必须配置风险档位表，至少包含：

- 名义价值上限；
- 最大杠杆；
- 初始保证金率（IMR）；
- 维持保证金率（MMR）；
- 清算手续费率；
- 强平分段目标档位。

### 10.5 平均开仓价与部分平仓

- 加仓时采用加权平均开仓价；
- 部分平仓时按已关闭数量结转 realized PnL；
- 减仓不能影响未关闭部分的平均开仓价规则；
- 清算产生的成交同样视作 fill，必须进入仓位与账本。

### 10.6 用户可见资金字段映射

为了兼容原始文档要求，对外界面可展示为：

- **余额**：以 `可交易余额` 为主；
- **占用保证金**：`订单冻结 + 持仓保证金需求`；
- **冻结资金**：`提现冻结 + 人工审核冻结 + 系统安全冻结`；
- **权益**：必须单独展示，不能缺失；
- **未实现 PnL**：必须单独展示，不能混入余额。

## 11. 风控、强平、保险基金与 ADL

### 11.1 风控分层

#### 11.1.1 预交易风控

- 余额 / 保证金是否充足；
- 用户是否触发限仓；
- symbol 是否可交易；
- 行情是否新鲜；
- 单笔最大下单量是否超限；
- 当前净敞口是否允许该方向新增仓位；
- 是否触发风控黑名单/频率限制。

#### 11.1.2 运行时风控

- 实时重算用户风险率；
- 监控 symbol 级别净敞口；
- 监控行情源分歧、延迟、丢包；
- 监控热钱包余额、提现异常、归集滞后；
- 监控保险基金余额与坏账暴露。

### 11.2 清算触发条件

当账户风险率低于阈值时触发清算。建议按照以下顺序执行：

1. 冻结该账户新增开仓；
2. 取消相关挂单；
3. 释放订单冻结；
4. 尝试部分强平，降到更低风险档位；
5. 仍不满足维持保证金则执行全平；
6. 计算残余保证金并退回；
7. 若出现负权益，则由保险基金兜底；
8. 保险基金不足时触发 ADL。

### 11.3 清算执行要求

- 清算执行器必须与普通成交路径分离；
- 清算订单具有更高优先级；
- 清算期间该用户不可再提交增加风险的订单；
- 必须记录清算前后价格快照、仓位快照、保证金快照；
- 清算过程必须幂等，重复触发不会重复扣款。

### 11.4 部分强平

生产系统必须优先支持**部分强平**，而不是直接一刀切全平。目标是：

- 先将用户风险降至下一档可接受水平；
- 减少不必要的价格冲击；
- 降低坏账与用户体验损失。

### 11.5 保险基金（原文未写明，本版新增）

保险基金必须作为独立账户存在，来源包括：

- 平台注资；
- 部分强平罚金；
- 预留风险准备金；
- 经审批后转入的经营收益。

用途：

- 弥补穿仓或负权益；
- 缓冲极端行情下的坏账；
- 在 ADL 之前作为第一兜底层。

### 11.6 ADL（原文未写明，本版新增）

当满足以下条件时触发 ADL：

- 用户穿仓；
- 保险基金不足以覆盖坏账；
- 市场流动性或价格异常导致无法正常平掉风险。

ADL 要求：

- 仅作为最后兜底，不得作为常规风控手段；
- 必须有可解释的排序规则（如收益与杠杆综合排序）；
- 必须通知受影响用户；
- 必须在 Explorer 与后台中可查询。

### 11.7 风险熔断与 kill switch

必须支持以下总闸：

- 全站暂停开仓；
- 单 symbol 暂停开仓；
- 单方向暂停开仓；
- 强制 `REDUCE_ONLY`；
- 暂停提现；
- 暂停充值入账；
- 暂停对冲；
- 进入手工值守模式。

## 12. 资金费率与融资规则

### 12.1 Crypto 资金费率规则

对于同时存在于 Binance 与 Hyperliquid 的 Crypto 合约，默认采用：

1. 读取两个来源的 funding rate；
2. 归一到统一小时口径；
3. 做等权平均（权重可配置）；
4. 再施加 cap/floor；
5. 按小时批次结算。

建议公式：

```text
hourly_rate = clamp(avg(normalized_valid_source_rates), -funding_cap, funding_cap)
funding_fee = position_notional_at_settlement × hourly_rate
```

强制要求：

- 若只有单源有效，可走单源兜底；
- 若无有效源且超过降级阈值，禁止新增该 symbol 风险；
- 资金费结算价格必须使用抗操纵价格（建议 Index / Oracle 口径）。

### 12.2 Commodities / Equities 融资规则（原文缺失，本版新增）

对于非 Crypto 的 CFD，资金费率不能简单照搬 Crypto 逻辑。必须支持按资产类别配置融资模型，例如：

- 基准利率；
- 平台持仓融资点差；
- 借券成本 / 库存成本；
- 分红调整；
- 节假日 / 周末倍数。

因此 `funding_model` 至少支持：

- `EXTERNAL_AVG`：外部资金费率平均；
- `INTERNAL_CARRY`：内部融资模型；
- `FIXED_RATE`：固定费率；
- `DISABLED`：不结算资金费，仅用于特定产品。

### 12.3 结算批次要求

每个资金费率批次必须记录：

- `funding_batch_id`
- `symbol`
- `time_window`
- `source_rates`
- `normalized_rate`
- `cap_applied`
- `settlement_price`
- `affected_positions`
- `ledger_tx_ids`
- `status`

必须支持：

- 预览；
- 批次重试；
- 批次幂等；
- 批次回滚（通过冲正分录）。

### 12.4 资金费异常处理

- 批次生成失败不得部分落账；
- 同一批次重复执行不得重复结算；
- 若某一用户扣费后触发风险率跌破阈值，应进入清算流程；
- 资金费历史必须可查询并支持导出。

## 13. 平台净敞口与外部对冲系统（生产级新增）

### 13.1 为什么必须新增

CFD 模型下，平台是用户的直接对手方。若没有净敞口管理与对冲能力，平台将裸露承受用户盈亏波动，无法称为生产级设计。因此该模块为**生产强制能力**。

### 13.2 净敞口计算

必须至少按以下维度计算净敞口：

- symbol；
- 资产类别；
- 多/空方向；
- 名义价值；
- Delta（可简化为线性敞口）；
- 上游对冲覆盖率。

示例：

```text
Net Exposure(symbol) = Σ用户多仓名义价值 - Σ用户空仓名义价值 - Σ外部对冲头寸
```

### 13.3 软/硬阈值

每个 symbol 必须配置：

- `soft_limit`：达到后扩大点差、降低杠杆、限制单笔规模；
- `hard_limit`：达到后禁止继续增加该方向风险；
- `panic_limit`：达到后强制 reduce-only 或自动对冲。

### 13.4 对冲执行

若启用外部对冲：

- 对冲动作必须走独立连接器；
- 上游账户资产单独建账；
- 对冲订单、对冲成交、对冲手续费、对冲 PnL 全部可追溯；
- 对冲失败必须触发风险告警；
- 对冲系统异常时，平台应自动收紧或关闭高风险方向开仓。

### 13.5 对冲账户对账

必须对以下三份数据做三方核对：

- 内部 `HEDGE_*` 账本；
- 上游账户余额 / 持仓；
- 风险敞口引擎视图。

- `chain_tx_hash`

1. **链上充值对账**：监听记录 vs 链上真实 transfer；

5. **账本守恒对账**：按资产维度检查守恒；
6. **资金费批次对账**：批次明细 vs FUNDING_POOL；
7. **仓位对账**：持仓视图 vs fill 明细重放。

### 14.5 导出与归档

- Explorer 查询结果支持 CSV/JSON 导出；
- 审计日志与资金流水必须长期归档；
- 每日生成账本快照、资金快照、仓位快照；
- 关键归档文件需要校验和与时间戳。

## 17. 非功能要求（高可用、性能、安全、监控、灾备）

### 17.1 性能目标

以下为建议的生产基线目标：

- 下单接口：正常负载下 `p95 < 20ms`，`p99 < 50ms`（不含外网 RTT）；
- 撤单接口：`p99 < 30ms`；
- 订单到成交回执：`p99 < 100ms`；
- 风险率重算：热点账户在价格更新后 `1s` 内完成；
- 清算触发到执行入队：`< 500ms`；
- WebSocket 账户变更推送：`p99 < 1s`；
- Explorer 常规查询：`p95 < 200ms`；
- 达到确认数后的充值入账：`< 60s`；
- 提现广播：通过审核后 `p95 < 120s`。

### 17.2 可用性目标

- 用户交易 API 月可用性建议不低于 `99.95%`；
- 风险引擎、账本服务、提现服务不允许单点；
- 行情至少双源；
- 关键数据库必须主从或多副本；
- 任何单服务故障不得导致账本错账。

### 17.3 RPO / RTO

建议目标：

- `RPO <= 5 分钟`；
- `RTO <= 30 分钟`；
- 账本、订单、仓位、配置库必须纳入灾备；
- 至少季度进行一次恢复演练。

### 17.4 安全要求

必须至少支持：

- TLS 全链路传输；
- 数据库加密或磁盘加密；
- 密钥存储在 KMS/HSM/专用签名服务；
- 管理后台 MFA；
- RBAC + 审计；
- 限流、防重放、防刷单；
- DDoS / WAF 预留；
- 依赖漏洞扫描与镜像扫描；
- 敏感字段脱敏；
- 行为风控与异常登录告警；
- 提现高危操作双重校验。

### 17.5 监控与告警

必须至少监控：

- 账本守恒异常；
- 余额缓存与账本偏差；
- 行情源心跳与延迟；
- 标记价与指数价偏离；
- 订单拒单率；
- 清算队列长度；
- 保险基金余额；
- 热钱包余额；
- 提现失败率；
- 对冲失败率；
- 配置变更频率；
- 数据库复制延迟；
- 消息积压；
- RPC 节点异常。

### 17.6 日志与链路追踪

- 所有写路径必须带 `trace_id`；
- 订单、成交、账本、清算、提现全链路可串联；
- 日志必须结构化；
- 用户级日志需可按 `address / order_id / event_id` 检索；
- 审计日志与业务日志分离存储。

### 17.7 数据保留策略

建议：

- 账本、审计日志：长期保留；
- 订单与成交：长期保留；
- 原始行情：热存 7~30 天，冷存按需归档；
- 链路追踪：7~30 天；
- 报表快照：按日归档。

## 19. 验收标准（建议作为交付门槛）

### 19.1 资金与账本验收

- 任意业务链路完成后，账本按资产维度守恒；
- 重复请求不会重复记账；
- 不存在绕过账本修改余额的路径；
- 任一笔充值、提现、转账、成交、资金费、清算都可从 Explorer 回溯到账本明细。

### 19.2 交易与风险验收

- 市价 / 限价 / TP / SL / reduce-only 均可闭环运行；
- 全仓 + 双向持仓可用；
- 清算可自动触发并落账；
- 保险基金与 ADL 路径经过演练；
- 行情降级时系统可自动进入受限模式。

### 19.3 运维与审计验收

- 管理后台操作均有审计；
- 配置中心支持审批与回滚；
- 至少一套钱包对账、一套账本守恒对账、一套对冲对账可执行；
- 关键告警可触达并可演练；
- review 环境支持 faucet，但 prod 环境严格关闭。

## 20. 本版新增的生产级能力清单

相较原始文档，本版补充并明确了以下**生产级必需功能**：

1. **完整提现系统**（状态机、审核、冻结、广播、确认、回滚）；

7. **限价单在 CFD 模式下的明确语义**；
8. **行情熔断、降级、symbol reduce-only 模式**；
9. **Commodities / Equities 的时段、停牌、分红、拆股、展期处理**；
10. **配置中心版本化、审批、回滚**；
11. **审计日志、对账体系、补账规范**；
12. **docker compose review 模式与 prod 安全边界隔离**；
13. **测试水龙头账本化与环境隔离**；
14. **非功能指标、SLO、RPO/RTO、监控告警要求**；
15. **对冲、钱包、清算、资金费的批次幂等与回放能力**。

## 结论

原始文档已经给出了系统目标与核心方向，但仍属于“高层需求草案”。本扩充版将其收敛为一份可落地的生产级规格文档，重点补齐了：
