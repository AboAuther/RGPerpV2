# 环境配置与配置加载规范

## 1. 目标

本文档定义各进程的环境配置、加载顺序、默认行为与安全边界。

## 2. 配置原则

1. 所有进程只从配置层读取环境，不直接硬编码地址、密钥和阈值。
2. 不同环境必须显式区分：`dev`、`review`、`staging`、`prod`。
3. 配置缺失时，对资金、风控、提现相关能力必须 fail closed。
4. 配置的最终来源必须可审计，运行时动态配置与静态启动配置分离。

## 3. 配置分层

### 3.1 启动时静态配置

由环境变量和 `.env` 文件提供，适用于：

- 服务监听地址
- MySQL / Redis / RabbitMQ DSN
- RPC 节点
- Vault / RouterFactory 地址
- Hyperliquid API 凭证
- JWT 密钥
- 环境模式

### 3.2 运行时动态配置

由 `config_items` 表提供，适用于：

- 手续费
- 资金费率周期
- 风险阈值
- symbol 状态
- 提现限额
- 对冲阈值

## 4. 加载顺序

所有 Go 进程统一按以下顺序加载配置：

1. 编译时默认值
2. `deploy/env/common.env`
3. `deploy/env/<APP_ENV>.env`
4. 进程级环境变量覆盖
5. 启动后从 MySQL 拉取动态配置快照

优先级：

```text
env var > env file > code default
```

## 7. 安全要求

1. 生产环境禁止将密钥提交到仓库。
2. `prod` 环境必须禁用 faucet 与 mock 行情。
3. 若 JWT 密钥、RPC 地址、Vault 地址缺失，相关进程不得启动。
4. Hyperliquid 凭证不得出现在前端环境变量中。

## 8. 启动失败规则

以下情况必须直接启动失败：

- `MYSQL_DSN` 缺失
- `RABBITMQ_URL` 缺失
- `JWT_ACCESS_SECRET` 或 `JWT_REFRESH_SECRET` 缺失
- 某条启用链的 `RPC + USDC + VAULT` 配置不完整
- `APP_ENV=prod` 且 `REVIEW_FAUCET_ENABLED=true`
