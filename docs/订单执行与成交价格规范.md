# 订单执行与成交价格规范

## 1. 文档目标

本文档定义 CFD 模式下订单执行、成交价格、触发逻辑与并发控制规则，确保：

1. 风控优先于交易；
2. 成交价格可复现、可审计；
3. 限价单不以劣于用户价格成交；
4. 订单、成交、仓位、账本在事务上严格一致。

## 2. 订单类型范围

P0 必须支持：

- `MARKET`
- `LIMIT`
- `STOP_MARKET`
- `STOP_LIMIT`
- `TAKE_PROFIT_MARKET`
- `TAKE_PROFIT_LIMIT`
- `REDUCE_ONLY`

P0 可不实现但需预留：

- `TRAILING_STOP`
- `IOC`
- `FOK`
- `POST_ONLY`

## 3. 订单字段实施要求

订单必须至少包含：

- `order_id`
- `client_order_id`
- `user_id`
- `symbol`
- `side`
- `position_effect`
- `type`
- `price`
- `trigger_price`
- `qty`
- `filled_qty`
- `avg_fill_price`
- `reduce_only`
- `time_in_force`
- `max_slippage_bps`
- `status`
- `reject_reason`
- `created_at`
- `updated_at`

## 4. 市场数据输入

执行引擎只接受来自 `Market Data Service` 的标准化行情，不直接访问交易所原始响应。

标准输入字段：

- `best_bid`
- `best_ask`
- `impact_bid(qty)`
- `impact_ask(qty)`
- `index_price`
- `mark_price`
- `source_health`
- `market_ts`

## 5. Execution Price 规则

### 5.1 基础公式

买入：

```text
execution_price_buy =
impact_ask(qty)
× (1 + base_spread_bps + slippage_bps + exposure_adj_bps)
```

卖出：

```text
execution_price_sell =
impact_bid(qty)
× (1 - base_spread_bps - slippage_bps - exposure_adj_bps)
```

### 5.2 输入参数

- `base_spread_bps`：平台基础点差
- `slippage_bps`：根据下单规模计算的滑点
- `exposure_adj_bps`：根据平台净敞口调整的价格偏移

### 5.3 安全规则

- 价格必须基于健康行情源；
- 若 `market_ts` 超过新鲜度阈值，拒绝新开仓；
- 若 `impact_bid/impact_ask` 缺失，拒绝市价单与触发单成交。

## 6. 订单校验顺序

提交订单时必须按以下顺序同步校验：

1. 会话与鉴权
2. 账户状态
3. symbol 状态
4. 交易时段
5. 数量与价格精度
6. 最小名义价值
7. `reduce_only` 合法性
8. 可交易余额
9. 杠杆与风险档位
10. 净敞口阈值
11. 行情新鲜度
12. 幂等键
13. 限流

任一失败必须拒单，不得部分冻结保证金。

## 7. 各订单类型语义

## 7.1 MARKET

- 提交即尝试成交；
- 成交价由 `Execution Price` 决定；
- 若超出用户 `max_slippage_bps`，拒绝成交。

## 7.2 LIMIT

- 订单进入 `RESTING`；
- 当合成可成交价格满足用户条件时，进入执行；
- 最终成交价不得劣于用户限价。

### 7.2.1 买入限价

满足条件：

```text
execution_price_buy <= user_limit_price
```

### 7.2.2 卖出限价

满足条件：

```text
execution_price_sell >= user_limit_price
```

## 7.3 STOP_MARKET / STOP_LIMIT

- 默认触发参考价为 `Mark Price`
- 到达触发价后产生新的待执行订单
- 触发与成交必须分开记录

## 7.4 TAKE_PROFIT_MARKET / TAKE_PROFIT_LIMIT

- 默认触发参考价为 `Mark Price`
- 触发后默认生成 `reduce_only` 订单

## 7.5 REDUCE_ONLY

- 只允许降低现有风险；
- 若订单会增加风险或反向开仓，必须拒绝；
- reduce-only 平仓不得因为剩余数量计算错误而转成反向开仓。

## 8. 订单状态机

```text
NEW -> ACCEPTED -> RESTING / TRIGGER_WAIT / FILLED
NEW -> REJECTED
RESTING -> PARTIALLY_FILLED -> FILLED
RESTING -> CANCELED / SYSTEM_CANCELED / EXPIRED
TRIGGER_WAIT -> ACCEPTED(派生执行) -> FILLED / RESTING / REJECTED
```

## 9. 订单冻结与释放

### 9.1 开仓单冻结

冻结金额：

```text
hold_amount = order_initial_margin_requirement + estimated_open_fee + safety_buffer
```

### 9.2 平仓单冻结

- `reduce_only` 原则上不冻结保证金；
- 允许冻结预估手续费；
- 若会导致资金不足，仍可拒绝。

### 9.3 释放时机

- 用户撤单
- 系统撤单
- 订单过期
- 强平前系统撤销挂单
- 订单完全成交后释放剩余冻结

## 10. 部分成交

### 10.1 规则

- 一个订单可以生成多个 `fill`
- `filled_qty` 累加
- `avg_fill_price` 以加权平均计算

### 10.2 公式

```text
avg_fill_price_new =
(avg_fill_price_old × filled_qty_old + fill_price × fill_qty) / (filled_qty_old + fill_qty)
```

### 10.3 一致性要求

每个 `fill` 必须在单事务内完成：

- `fills` 插入
- `orders` 更新
- `positions` 更新
- `ledger` 更新
- `outbox` 更新

## 11. 开仓与平仓语义

### 11.1 开仓

- 无同方向仓位时创建仓位；
- 有同方向仓位时加仓；
- 有反方向仓位且非 hedge mode 时应先减仓；
- P0 强制支持 hedge mode，因此同 symbol 多空分别维护。

### 11.2 平仓

- 平仓按目标 side 对应仓位减少；
- 若 `reduce_only` 数量超过持仓数量，只允许平至零，不允许反向开仓；
- 部分平仓只结转已关闭数量的 realized pnl。

## 12. 幂等与并发控制

### 12.1 下单幂等

唯一键：

```text
user_id + client_order_id
```

### 12.2 撮合幂等

每个 fill 使用：

```text
fill_id
```

重复投递的执行事件命中相同 `fill_id` 时必须返回既有结果。

### 12.3 并发锁

提交订单与成交时需要锁定：

- 用户相关 `account_balance_snapshots`
- 用户 `positions`
- 目标 `orders`

推荐顺序：

1. `account_balance_snapshots`
2. `positions`
3. `orders`

按固定顺序加锁避免死锁。

## 13. 拒单条件

必须拒绝的场景：

- 行情源不健康
- 标记价冻结且 symbol 只允许 reduce-only
- 用户余额不足
- 风险档位超限
- `reduce_only` 试图扩大风险
- 价格精度错误
- 交易时段关闭
- 账户处于清算中
- 幂等键冲突且请求体不一致

## 14. 审计与可追溯

每个订单与成交必须保存：

- 下单参数
- 风险校验输入与输出
- 使用的价格快照
- 成交公式版本
- 费率配置版本
- trace_id

## 15. 禁止事项

- 禁止直接使用最新成交价作为成交基准；
- 禁止限价单以劣于用户价格成交；
- 禁止订单成功返回后异步再补写仓位或账本；
- 禁止 `reduce_only` 意外转成新开仓。
