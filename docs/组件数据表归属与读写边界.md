# 组件数据表归属与读写边界

## 1. 文档目标

本文档定义：

1. 每张核心表由哪个组件负责写入；
2. 哪些组件可读；
3. 哪些表属于强一致写链路；
4. 哪些表只能通过事件读模型更新。

本文件是对 [mysql-ddl.sql](/Users/xiaobao/RGPerp/spec/db/mysql-ddl.sql) 的组件归属补充。

## 2. 原则

1. 每张核心表必须有唯一主写组件。
2. 多组件可读，但不得多组件随意写。
3. 涉及资金、仓位、订单、清算的表，必须归属于强一致主写链路。
4. Explorer 和 Dashboard 优先使用读模型，不反向修改源表。

## 4. 强一致写链路表

以下表属于必须单事务提交的核心表：

- `ledger_tx`
- `ledger_entries`
- `account_balance_snapshots`
- `orders`
- `fills`
- `positions`
- `liquidations`
- `liquidation_items`
- `funding_batches`
- `funding_batch_items`
- `outbox_events`

说明：

- 不同业务场景涉及的表不完全相同；
- 但一旦某场景修改上述表，必须在单事务内完成。

## 5. 只允许通过特定组件写入的关键表

### 5.1 `ledger_tx` / `ledger_entries`

只能由 `Ledger Service` 写入。  
其他组件若有资金变化，必须调用 Ledger 域服务，不得直接插表。

### 5.2 `account_balance_snapshots`

只能由 `Ledger Service` 统一维护。  
Trade、Wallet、Funding、Liquidator 不得直接独立更新快照。

### 5.3 `orders`

只能由 `Order Service / Trade Engine / Liquidator` 在其授权流程内更新。  
Risk Engine 只能发事件，不直接改订单状态。

### 5.4 `positions`

只能由：

- Trade Engine
- Liquidator
- Funding Worker

按各自业务边界更新。  
Hedger 不得回写用户 `positions`。

### 5.5 `funding_batches`

只能由 Funding Worker 创建与推进。

### 5.6 `hedge_*`

只能由 Hedger 写入。  
Risk Engine 只负责生成 `hedge.requested`。

## 6. 读模型与源表分离

### 6.1 `explorer_events`

该表是读模型，不是资金或交易真相。  
来源：

- `outbox_events`
- RabbitMQ 事件消费
- 链上索引结果

禁止：

- 反向用 `explorer_events` 修正账本或订单源表

## 7. 并发控制边界

### 7.1 账户资金

涉及 `account_balance_snapshots` 的写操作必须加账户级锁。

### 7.2 仓位

涉及 `positions` 的写操作必须按 `user_id + symbol_id + side` 串行化。

### 7.3 订单

涉及 `orders` 的取消、成交、系统撤销必须保证同一订单串行更新。

## 8. 对账责任

### 8.1 Wallet Reconcile

负责：

- `vaults`
- `deposit_chain_txs`
- `withdraw_requests`
- 托管镜像账户

### 8.2 Ledger Reconcile

负责：

- `ledger_tx`
- `ledger_entries`
- `account_balance_snapshots`

### 8.3 Trade Reconcile

负责：

- `orders`
- `fills`
- `positions`

### 8.4 Funding Reconcile

负责：

- `funding_batches`
- `funding_batch_items`
- `FUNDING_POOL` 对应账务

### 8.5 Hedge Reconcile

负责：

- `hedge_orders`
- `hedge_fills`
- `hedge_positions`
- `HEDGE_*` 平台账户
