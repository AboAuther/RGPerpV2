# 实施前详细拆解与设计细化计划

## 1. 文档目的

本文档用于回答两个问题：

1. 在正式开始完整实现前，还需要把系统拆解到什么粒度。
2. 现有需求、架构、设计文档还应如何继续细化，才能让团队进入稳定开发状态。

## 2. 拆解原则

### 2.1 先冻结真相，再冻结接口

完整实现此类系统，必须先冻结以下三类“真相”：

1. 资金真相；
2. 交易真相；
3. 风险真相。

在这三类真相未冻结前，不应大规模开始页面与接口实现，否则后续重构成本极高。

### 2.2 先拆关键写链路，再拆读链路

优先拆解：

- 充值入账；
- 提现冻结与广播；
- 开仓；
- 平仓；
- 资金费率结算；
- 清算；
- 对冲。

读链路如 Explorer、Dashboard、报表可稍后细化。

### 2.3 先明确失败路径，再实现成功路径

这类系统的主要风险来自：

- 重复请求；
- 并发；
- 外部调用失败；
- 延迟；
- 重放；
- 重组；
- 程序重启。

因此每条核心链路都必须先补齐失败处理与恢复策略。

## 4. 开始编码前必须完成的五项细化

### 4.1 账本分录模板

要把以下动作逐项写清楚：

- deposit detected
- deposit credited
- withdraw hold
- withdraw broadcasted
- withdraw refunded
- transfer
- order margin hold
- order margin release
- open fill
- close fill
- trading fee
- funding payment
- liquidation
- insurance fund use
- hedge collateral adjustment

每项至少要明确：

- 业务前置条件；
- 影响的子账户；
- 分录方向；
- 幂等键；
- 回滚方式；
- Explorer 展示字段。

### 4.2 风险与清算公式

必须把以下公式写成规范，而不是散落在代码里：

- notional
- unrealized pnl
- realized pnl
- equity
- available balance
- initial margin requirement
- maintenance margin requirement
- margin ratio
- liquidation price
- bankruptcy price
- funding fee
- liquidation penalty

同时要为每个公式补充：

- 输入字段；
- 精度要求；
- 舍入规则；
- 异常保护；
- 示例。

### 4.3 Vault 合约设计

必须明确：

- 合约最小接口；
- 事件列表；
- 权限角色；
- pause 逻辑；
- 提现执行流程；
- 多链部署方式；
- 升级策略是否存在；
- Foundry 测试覆盖点。

### 4.4 API 与事件 Schema

当前 API 清单只有端点级别，下一步要补到 schema 级别：

- request body
- response body
- error code
- idempotency header
- auth requirement
- rate limit
- example payload

事件也要补到字段级：

- envelope
- producer
- routing key
- payload schema
- retry behavior
- dead-letter policy

### 4.5 DDL 与索引策略

要把当前实体表设计落成：

- 建表 SQL；
- 主键策略；
- 唯一键；
- 核心查询索引；
- 乐观锁或版本字段；
- 分库分表预留位；
- 清理与归档策略。

## 5. 编码前建议的进一步任务拆解

### 5.1 Workstream A：Contract + Wallet

#### 目标

让链上托管与链下索引闭环先成立。

#### 子任务

- Vault 接口定义
- Vault 事件定义
- Foundry 测试
- indexer 游标与确认机制
- deposit credit worker
- withdraw broadcaster

### 5.2 Workstream B：Ledger + Account

#### 目标

让所有资金流都有统一落账模型。

#### 子任务

- account subtype seed
- ledger transaction service
- ledger entry repository
- account snapshot updater
- outbox integration
- reconciliation hooks

### 5.3 Workstream C：Trade + Position

#### 目标

建立下单、成交、仓位三位一体的核心交易闭环。

#### 子任务

- symbol metadata
- market data cache
- order lifecycle
- fill generation
- position mutation
- polling refresh

### 5.4 Workstream D：Risk + Liquidation + Funding

#### 目标

建立风险闭环，确保系统不会因价格波动立即错账或失控。

#### 子任务

- pre-trade risk
- runtime risk snapshots
- liquidation trigger queue
- liquidator execution
- insurance fund bookkeeping
- funding scheduler and batch applier

### 5.5 Workstream E：Hedger + Ops

#### 目标

把平台敞口管理、Explorer、后台治理和对账补齐。

#### 子任务

- exposure aggregation
- Hyperliquid connector
- hedge intent lifecycle
- explorer read model
- admin audit
- reconciliation jobs

### 5.6 Workstream F：Frontend + Review Environment

#### 目标

把用户可见界面、后台界面与评审环境串起来。

#### 子任务

- auth pages
- trading terminal
- account center
- explorer UI
- admin UI
- docker compose review demo path

## 6. 建议的开发前顺序

### Phase A：规格冻结

必须先完成：

1. 分录模板；
2. 公式规范；
3. Vault 合约说明；
4. DDL；
5. OpenAPI 与事件 schema。

### Phase B：骨架搭建

并行推进：

1. 仓库目录；
2. 基础配置；
3. MySQL/Redis/RabbitMQ 连接层；
4. 统一日志、错误码、trace；
5. 合约工程和前端壳工程。

### Phase C：资金主链路

优先完成：

1. 登录；
2. 充值；
3. 提现；
4. 转账；
5. Explorer 资金视图。

### Phase D：交易主链路

依次完成：

1. 行情；
2. 下单；
3. 成交；
4. 仓位；
5. 手续费；
6. 轮询刷新与失败态。

### Phase E：风险主链路

依次完成：

1. 风险计算；
2. 清算；
3. 资金费率；
4. 对冲；
5. 审计与对账。

## 9. 结论

当前文档体系已经足以支撑“开始搭框架”，但还不足以支撑“完整稳定地实现整个项目”。  
真正缺的不是更多宏观描述，而是以下实施级细节：

1. 明确分录；
2. 明确公式；
3. 明确 contract ABI；
4. 明确 API/事件 schema；
5. 明确失败处理与恢复路径。

把这五层补齐之后，整个项目就会从“设计完成”进入“可持续交付”的状态。
