# CFD 永续合约交易所需求拆解与分阶段实施路线

## 1. 文档定位

本文档用于将当前题目要求与技术约束收敛为可执行的产品与系统实施基线，覆盖：

- 核心功能模块拆解；
- 端到端业务流程；
- 高风险环节识别；
- 必须确认问题清单；
- 面向当前交付目标的分阶段实施路线。

本文档以 [requirements/技术设计文档.md](/Users/xiaobao/RGPerp/requirements/技术设计文档.md) 为强制约束基线，并在不冲突前提下参考 [docs/CFD永续合约交易所_扩充版系统需求与技术设计.md](/Users/xiaobao/RGPerp/docs/CFD永续合约交易所_扩充版系统需求与技术设计.md) 补齐生产交付所需能力。

## 2. 必须确认问题

以下问题若不先确认，将直接影响账本口径、风控策略、交付边界和验收结果。

### 2.1 产品边界与交易标的

1. “两平台均存在的合约交易对”默认是否采用 Binance 与 Hyperliquid 作为首期价格源。
2. 首期是否要求同时交付 Crypto、Commodities、Equities 三类资产，还是允许以 Crypto 为 P0，其他资产类别完成元数据与扩展设计但不全部上线。
3. Commodities 与 Equities 首期是否要求真实交易时段、停牌、分红、拆股、展期规则落地，还是允许保留规则引擎接口并通过静态配置模拟。

### 2.2 价格与成交语义

1. CFD 模式下限价单是否采用“满足价格条件后，由执行引擎以不劣于限价的价格成交”的语义。
2. 成交价格中的“平台滑点参数”是否允许按 symbol、方向、订单规模、平台净敞口分别配置。
3. 止盈止损触发参考价是否统一使用 Mark Price，是否允许按 symbol 切换为 Index Price。

### 2.3 保证金与风险规则

1. 首期是否强制只支持 Cross Margin，不交付 Isolated Margin。
2. 强平是否要求首期就支持部分强平；若交付时间极紧，是否允许首版先落地全平，但设计上预留部分强平。
3. 穿仓场景是否接受用户余额为负；若接受，负权益后的保险基金兜底与后续追偿是否纳入首期。
4. 维持保证金“参考标准上浮固定比例值”的参考标准来源是否由系统内 Tier 表配置，不依赖外部平台实时拉取。

### 2.4 资金费率与融资规则

1. Funding Rate 首期是否仅对 Crypto 生效，Commodities 与 Equities 是否采用 `DISABLED` 或 `INTERNAL_CARRY` 占位模型。
2. 资金费率平均逻辑在单源缺失、双源分歧过大、无有效源时的降级策略是否允许采用“禁止新开仓、允许减仓/平仓”。

### 2.5 充值、提现与钱包

1. “为每个用户分配独立充值地址”首期是否为每链唯一地址，还是允许通过 memo/内部映射替代。
2. 提现模块虽未在原始文档展开，但账务类型已包含 `WITHDRAW`。首期是否按生产要求纳入完整状态机、审核、广播、确认与失败冲正。
3. review 环境中的“增加测试资金”是否采用链上 mock 充值、系统 faucet 接口，或两者同时提供。

### 2.6 审计、后台与验收

1. Explorer 的“按区块查询”是否明确为仅查询链上充值/提现事件，不适用于纯链下交易事件。
2. 数据看板首期是否只要求只读查询，还是要求具备部分运营操作联动。
3. 配置中心是否要求双人审批与版本回滚在首期全部完成，还是允许 review 版本先完成版本化与审计，审批链简化。

## 3. 推荐实施方案

### 3.1 推荐方案

面向当前交付目标，推荐采用“模块化单体 + 强一致数据库事务 + Outbox 异步事件”的实现方案。

### 3.2 推荐理由

1. 当前交付周期短，系统复杂度高，先保证账本、交易、仓位、风控在同一事务边界内正确闭环，比过早拆微服务更重要。
2. 统一使用 PostgreSQL 承担订单、仓位、账本、配置、审计核心写入，可显著降低跨服务分布式事务风险。
3. Redis 用于 nonce、会话、限流、热点价格缓存；消息总线用于 Explorer、看板、通知、对账等最终一致链路。
4. 模块化单体仍可按领域边界组织为 Auth、Ledger、Wallet、Market、Order、Position、Risk、Funding、Explorer、Admin 等独立模块，后续可平滑拆分。
5. 该方案最适合本题“生产级 correctness 优先、docker compose 可启动、50 小时内可交付”的目标。

### 3.3 不推荐方案

- 不推荐以纯前后端 demo 方式直接修改余额或绕过账本完成交易闭环。
- 不推荐首期即做高耦合微服务网格，否则大概率把时间消耗在分布式一致性、联调和部署问题上。
- 不推荐将撮合、账本、仓位、清算拆成跨库异步写入链路，否则高风险场景下极易出现错账与重复扣费。

## 4. 核心业务目标与系统不变式

### 4.1 核心业务目标

1. 建立以统一账本为中心的资金系统。
2. 建立基于外部参考价格的 CFD 永续合约交易闭环。
3. 建立围绕账户权益、维持保证金率、强平、资金费率的风险控制体系。
4. 建立支持链上充值、提现、归集、对账、审计、后台与数据看板的生产级运行基础。

### 4.2 核心不变式

1. 所有资金变动必须经由账本事务落账。
2. 同一 `ledger_tx_id + asset` 下所有分录金额之和必须为零。
3. 任何余额快照都只是账本结果，不得独立成为资金真相。
4. 订单状态、仓位变化、账本变化必须在关键写链路内原子提交。
5. 风控优先于交易体验，行情异常、价格失真、风险率过低时必须拒绝开仓或触发降级。
6. 所有关键操作必须可审计、可回放、可对账。

## 5. 核心功能模块拆解

以下模块按照技术设计文档强制要求覆盖。

### 5.1 账户、认证与权限模块

#### 模块职责

- EVM 地址登录；
- nonce 挑战、签名验签、防重放；
- access token / refresh token 会话管理；
- 账户状态管理；
- 前台与后台权限隔离。

#### 核心能力

- 一个地址对应一个主账户；
- 支持账户状态 `ACTIVE`、`TRADE_ONLY`、`WITHDRAW_ONLY`、`FROZEN` 等；
- 后台角色至少覆盖运营、风控、财务、审计、系统管理员。

#### 关键约束

- nonce 只能使用一次；
- 登录消息必须绑定域名、环境、过期时间；
- 高危操作应具备二次确认。

### 5.2 统一账本与资金子账户模块

#### 模块职责

- 维护统一账本事务头与分录明细；
- 维护用户与平台账务子账户；
- 维护余额快照与幂等控制；
- 提供补账、冲正、守恒校验能力。

#### 推荐账户模型

- 用户侧：`USER_WALLET`、`USER_ORDER_MARGIN`、`USER_POSITION_MARGIN`、`USER_WITHDRAW_HOLD`
- 平台侧：`SYSTEM_POOL`、`TRADING_FEE_ACCOUNT`、`WITHDRAW_FEE_ACCOUNT`、`PENALTY_ACCOUNT`、`FUNDING_POOL`、`INSURANCE_FUND`
- 托管侧：`CUSTODY_HOT`、`CUSTODY_WARM`、`CUSTODY_COLD`、`DEPOSIT_PENDING_CONFIRM`、`WITHDRAW_IN_TRANSIT`

#### 关键约束

- 禁止直接更新余额绕过账本；
- 每笔业务操作使用唯一 `ledger_tx_id`；
- 补账必须使用冲正或补偿分录。

### 5.3 充值、提现、转账与钱包模块

#### 模块职责

- 多链 USDC 充值监听与确认入账；
- 提现申请、冻结、风控审核、广播、确认、失败冲正；
- 内部账户间转账；
- 热温冷钱包管理与归集；
- review 环境 faucet。

#### 核心能力

- 支持 Ethereum、Arbitrum、Base；
- 仅识别白名单 USDC 合约；
- 充值必须等待确认数；
- 提现必须经过冻结和状态机；
- 内转实时到账且不收费；
- 所有链上链下动作必须可回溯到账本与 Explorer。

### 5.4 交易对与产品元数据模块

#### 模块职责

- 维护交易对、分类、精度、最小下单单位、风险档位、交易时段、价格源映射；
- 管理 symbol 状态、上线、下线、熔断、reduce-only。

#### 核心能力

- 资产类别支持 Crypto、Commodities、Equities；
- symbol 映射支持多源命名差异；
- 各 symbol 独立配置杠杆、IMR、MMR、滑点、手续费、资金费模型。

### 5.5 行情与定价模块

#### 模块职责

- 接入多源行情；
- 进行标准化、健康检查、时延监控；
- 生成 Index Price、Mark Price、Execution Price；
- 支持异常剔除、熔断、降级。

#### 核心能力

- 主源建议 Binance + Hyperliquid；
- 标记价格用于未实现 PnL、清算、风控、TP/SL；
- 成交价格基于外部深度聚合价与平台滑点参数；
- 异常行情下支持暂停开仓与 `REDUCE_ONLY`。

### 5.6 订单与执行模块

#### 模块职责

- 处理市价单、限价单、止盈止损；
- 完成下单校验、挂单、触发、成交、撤单；
- 生成订单状态、成交记录与价格快照关联。

#### 核心能力

- 市价单；
- 限价单；
- `STOP` / `TAKE_PROFIT`；
- `REDUCE_ONLY`；
- 订单冻结、部分成交、成交拆分；
- 幂等下单与客户端订单号支持。

#### 关键约束

- 开仓单必须冻结保证金与预估手续费；
- 清算前必须先撤销相关挂单并释放冻结；
- CFD 限价单成交价不得劣于用户限价。

### 5.7 仓位、保证金与 PnL 模块

#### 模块职责

- 维护双向持仓；
- 计算仓位名义价值、初始保证金、维持保证金、未实现与已实现 PnL；
- 生成 liquidation price、bankruptcy price 等派生值。

#### 核心能力

- Cross Margin；
- Hedge Mode；
- `user + symbol + side` 维度维护仓位；
- 部分平仓按已平数量结转 realized PnL。

### 5.8 风控、清算、保险基金与 ADL 模块

#### 模块职责

- 预交易风控；
- 运行时账户风险率重算；
- 自动强平与罚金处理；
- 保险基金兜底；
- 极端情形 ADL。

#### 核心能力

- 账户风险率监控；
- 基于 Mark Price 触发清算；
- 支持风险参数动态配置；
- 清算事件可追溯；
- 保险基金独立建账；
- 为 ADL 预留规则与展示面。

### 5.9 资金费率与融资模块

#### 模块职责

- 采集外部 funding 数据；
- 归一化为小时口径；
- 生成资金费批次；
- 按仓位结算资金费并落账。

#### 核心能力

- Crypto 默认取 Binance 与 Hyperliquid 平均值；
- 单源缺失可降级；
- 结算从 `USER_POSITION_MARGIN` 扣收或返还；
- 批次必须幂等、可查询、可导出、可冲正。

### 5.10 Explorer、审计与对账模块

#### 模块职责

- 提供事件查询、账本映射、链上链下关联；
- 沉淀管理员操作审计；
- 提供链上充值、提现、账本守恒、仓位、资金费等对账任务。

#### 核心能力

- 查询维度覆盖 `event_id`、`ledger_tx_id`、`chain_tx_hash`、`order_id`、`fill_id`、`position_id`、`funding_batch_id`、`address`、`block_height`；
- 支持 CSV/JSON 导出；
- 支持快照归档。

### 5.11 数据看板与后台管理模块

#### 模块职责

- 提供资金看板、风险看板、交易看板；
- 提供用户查询、提现审核、风险参数调整、补账审批、任务重试等后台能力。

#### 核心能力

- 账本各项数据查询；
- 多空分布、净敞口、风险快照；
- 清算、资金费、钱包、提现队列监控。

### 5.12 配置中心与变更治理模块

#### 模块职责

- 动态管理杠杆、手续费、滑点、罚金、资金费周期、维持保证金上浮比例等关键参数；
- 提供版本化、生效时间、审批、回滚与审计。

#### 关键约束

- 风险与提现相关配置不应直接裸改；
- 所有变更必须有前后值、操作人、审批人、原因、可回滚能力。

### 5.13 非功能与运行保障模块

#### 模块职责

- 高可用；
- 性能保障；
- 安全控制；
- 监控告警；
- 容灾恢复；
- docker compose review 环境。

#### 核心能力

- 强一致写链路与最终一致读链路分离；
- 核心数据持久化；
- 结构化日志、trace_id、链路追踪；
- 风险熔断和 kill switch；
- review 环境与生产环境安全隔离。

## 6. 完整业务流程

### 6.1 登录与会话流程

1. 用户请求 nonce。
2. 服务端生成一次性 nonce、绑定域名、链 ID、过期时间。
3. 用户使用 EVM 钱包签名。
4. 服务端校验签名、nonce、环境绑定与防重放。
5. 创建账户或识别已有账户。
6. 返回 access token / refresh token。
7. 记录登录审计、设备指纹与 IP。

### 6.2 充值入账流程

1. 用户查询充值地址并向链上地址转入 USDC。
2. 钱包监听器发现链上事件，状态记为 `DETECTED`。
3. 达到确认数前，资产记入 `DEPOSIT_PENDING_CONFIRM`，不可作为可交易余额。
4. 确认数达标后生成充值业务事件与账本事务。
5. 从待确认镜像账户转入用户 `USER_WALLET`。
6. Explorer 建立链上 `tx_hash` 与内部 `ledger_tx_id` 映射。
7. 归集任务将资金归集至热/温钱包，并同步更新托管镜像账户。
8. 对账任务校验链上资产与账本镜像一致。

### 6.3 提现流程

1. 用户提交提现申请。
2. 系统校验账户状态、余额、限额、地址风险、黑名单等规则。
3. 系统冻结用户资金：`USER_WALLET -> USER_WITHDRAW_HOLD`。
4. 命中审核规则时进入 `RISK_REVIEW`，否则进入签名/广播流程。
5. 钱包服务签名并广播链上交易。
6. 广播成功后将冻结资金转入 `WITHDRAW_IN_TRANSIT`，并记录提现手续费。
7. 链上确认完成后更新托管镜像账户并将提现单置为 `COMPLETED`。
8. 广播失败或链上失败时必须通过冲正分录释放冻结并恢复用户可用余额。

### 6.4 内部转账流程

1. 用户提交目标地址与金额。
2. 系统校验双方账户状态、黑名单与频率限制。
3. 单个账本事务完成转出账户扣减与转入账户增加。
4. 双方账户实时可见。
5. Explorer 和审计日志可查询。

### 6.5 下单与挂单冻结流程

1. 用户提交订单。
2. 系统执行账户状态、symbol 状态、交易时段、价格精度、最小名义价值、杠杆、限仓、净敞口、行情新鲜度、幂等与限流校验。
3. 开仓单冻结保证金与预估手续费：`USER_WALLET -> USER_ORDER_MARGIN`。
4. 市价单进入执行引擎；限价/触发单进入挂单或等待触发池。
5. 系统向用户返回订单状态。

### 6.6 成交与开仓流程

1. 执行引擎读取最新可用价格源，生成 Execution Price。
2. 订单被全部或部分成交，生成 `fill`。
3. 系统在同一强一致写链路内更新订单状态、仓位状态、账本分录。
4. 对于开仓成交：
   - `USER_ORDER_MARGIN` 扣减；
   - `USER_POSITION_MARGIN` 增加；
   - 手续费进入 `TRADING_FEE_ACCOUNT`。
5. 更新仓位均价、名义价值、保证金需求与用户权益。
6. 通过 WebSocket 推送订单、成交、仓位与账户变化。

### 6.7 平仓流程

1. 用户提交平仓单或 reduce-only 单。
2. 执行引擎生成平仓成交。
3. 系统按已平数量计算 realized PnL。
4. 账本将 `USER_POSITION_MARGIN` 转回 `USER_WALLET`，同时将盈亏与手续费分别记入 `SYSTEM_POOL` 和 `TRADING_FEE_ACCOUNT`。
5. 若仓位完全关闭，则更新仓位为关闭状态；若部分平仓，则保留剩余仓位并更新数量。

### 6.8 资金费率结算流程

1. Funding 服务按周期拉取 Binance 与 Hyperliquid funding 数据。
2. 数据标准化为统一小时口径并完成有效性检查。
3. 生成资金费率批次并锁定结算价格与影响仓位集合。
4. 对每个持仓计算 `funding_fee = position_notional × funding_rate`。
5. 系统以批次方式更新 `USER_POSITION_MARGIN` 与 `FUNDING_POOL`。
6. 批次落账后写入历史记录、Explorer 和报表。
7. 若结算导致账户风险率跌破阈值，立即移交清算引擎。

### 6.9 清算流程

1. 风控引擎实时基于 Mark Price 重算账户风险率。
2. 风险率触达阈值后冻结该账户新增开仓能力。
3. 取消相关挂单并释放 `USER_ORDER_MARGIN`。
4. 清算执行器优先尝试部分强平，将仓位降至更低风险档位。
5. 若仍不满足维持保证金要求，则执行全平。
6. 根据清算成交结果，将持仓保证金分配至 `SYSTEM_POOL`、`PENALTY_ACCOUNT` 与用户 `USER_WALLET`。
7. 若出现负权益，由 `INSURANCE_FUND` 兜底；保险基金不足时进入 ADL。
8. 记录清算前后价格快照、账户快照、仓位快照、账本事务与清算日志。

### 6.10 Explorer 与对账流程

1. 业务写链路成功后，通过 outbox 异步投递事件。
2. Explorer 服务消费事件，建立多维索引。
3. 用户或后台可按 `tx_hash`、地址、订单、仓位、时间、区块查询。
4. 对账任务周期性执行链上对账、账本守恒校验、资金费批次校验、仓位重放校验。
5. 异常任务进入告警与人工处理流程。

### 6.11 配置变更流程

1. 管理员提交参数变更。
2. 系统校验权限、作用域与生效方式。
3. 高风险配置进入审批链；低风险配置可即时生效。
4. 配置写入版本库并产生审计记录。
5. 相关服务订阅配置变更事件并按安全点加载。
6. 若变更引发异常，可按版本回滚。

## 7. 高风险环节

### 7.1 资金与账本一致性风险

- 直接修改余额、漏记系统账户、手续费或罚金分录不完整，将直接导致错账。
- 订单、仓位、账本跨事务写入会产生“成交成功但账务失败”或“账务成功但仓位未更新”问题。
- 充值确认、提现广播、资金费批次、清算重复执行若无幂等设计，将产生重复记账。

### 7.2 价格与清算风险

- 若 Mark Price 可被单源或最新成交价操纵，将误触发止损与强平。
- 行情源延迟、异常偏离或短时中断若未触发降级，会造成错误成交与系统性穿仓。
- CFD 限价单语义不清会导致“触发了但成交价比限价更差”的争议。

### 7.3 风控与穿仓风险

- 仅做全平不做部分强平时，极端行情下冲击更大，坏账扩散风险更高。
- Commodities / Equities 若忽略交易时段、停牌、展期、分红，会造成价格模型失真与错误风险评估。
- 若不做平台净敞口管理，平台可能在单边行情中承担过大系统性风险。

### 7.4 钱包与链上风险

- 充值监听未处理重组、重复回调、错误 token、错误链，会导致误入账或漏入账。
- 提现若无审核、限额、热钱包阈值与失败冲正，将形成直接资金损失风险。
- review faucet 若未做环境隔离，存在污染生产环境风险。

### 7.5 安全与权限风险

- EVM 登录若 nonce 可复用或签名消息未绑定域名，存在重放攻击风险。
- 管理后台若缺乏 RBAC、MFA、审计和双人复核，高危资金与风控操作不可控。
- 配置中心若支持即时裸改风控参数，可能在运行中引入大面积错清算。

### 7.6 交付风险

- 以生产级要求审视，本项目真正难点不在页面或接口数量，而在账务一致性、精度、幂等、回放、清算和可恢复性。
- 若前期不先固化领域模型、状态机、账本分录模板与公式，后续联调会持续返工。

## 11. 结论

本项目的实现主线应当明确为：

1. 先用统一账本建立资金真相；
2. 再用价格、订单、仓位和风控建立交易真相；
3. 最后用对账、审计、配置和监控建立运行真相。

在当前交付目标下，最优策略不是追求服务拆分数量或界面数量，而是优先确保以下四件事成立：

1. 不错账；
2. 不重复扣费；
3. 不因行情异常错误清算；
4. 任何关键事件都能追溯与恢复。

满足上述四点后，再逐步补齐对冲、ADL、复杂资产类别规则和高级运营能力，能够在技术要求范围内实现更稳健的生产级交付。
