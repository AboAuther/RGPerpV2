# 风险计算与清算公式规范

## 1. 文档目标

本文档定义交易、保证金、风险率、清算、资金费率相关计算规则，确保：

1. 风控优先于交易；
2. 计算公式一致；
3. 实现可审计、可复算；
4. 所有边界条件显式定义。

## 2. 总体原则

### 2.1 风险真相

- 清算触发以账户级风险率为准；
- `liquidation_price` 为展示与预警字段，不作为唯一触发依据；
- 任何单一派生字段错误不得导致系统忽略真实风险。

### 2.2 价格优先级

- 账户风险、清算、TP/SL 默认使用 `Mark Price`
- Funding 结算价格使用 `Index Price` 或配置指定的防操纵价格
- 成交结算使用 `Execution Price`

### 2.3 精度

- 所有公式采用 `DECIMAL(38,18)` 计算
- 展示层可降为 6~8 位，但服务内不得截断
- 每个公式必须先计算后舍入，不得中间提前截断

## 3. 基础定义

### 3.1 方向定义

- LONG：`direction_sign = +1`
- SHORT：`direction_sign = -1`

### 3.2 合约乘数

P0 默认：

```text
contract_multiplier = 1
```

若 symbol 配置特殊乘数，必须从元数据读取，不允许硬编码。

### 3.3 基础输入字段

- `qty`
- `avg_entry_price`
- `mark_price`
- `execution_price`
- `contract_multiplier`
- `imr`
- `mmr`
- `liquidation_fee_rate`
- `funding_rate`

## 4. 核心公式

### 4.1 持仓名义价值

```text
notional = abs(qty) × mark_price × contract_multiplier
```

#### 异常保护

- `mark_price <= 0` 时，禁止更新风险状态；
- 若 `qty = 0`，则 `notional = 0`。

### 4.2 未实现 PnL

```text
unrealized_pnl = direction_sign × qty × (mark_price - avg_entry_price) × contract_multiplier
```

说明：

- LONG：价格上涨为正；
- SHORT：价格下跌为正。

### 4.3 已实现 PnL

对于部分平仓数量 `closed_qty`：

```text
realized_pnl_delta = direction_sign × closed_qty × (close_price - avg_entry_price) × contract_multiplier
```

累加规则：

```text
realized_pnl = realized_pnl_old + realized_pnl_delta
```

### 4.4 初始保证金需求

```text
initial_margin_requirement = notional × imr
```

### 4.5 维持保证金需求

```text
maintenance_margin_requirement = notional × mmr
```

### 4.6 预估手续费占用

开仓时预冻结：

```text
estimated_open_fee = order_notional × taker_fee_rate_ceiling
```

平仓 reduce-only 订单可不冻结保证金，但建议预估手续费：

```text
estimated_close_fee = order_notional × taker_fee_rate_ceiling
```

### 4.7 账户抵押余额

```text
collateral_balance =
USER_WALLET
+ USER_ORDER_MARGIN
+ USER_POSITION_MARGIN
+ USER_WITHDRAW_HOLD
```

### 4.8 账户权益

```text
equity =
collateral_balance
+ Σ unrealized_pnl
- Σ pending_trading_fee
- Σ pending_funding_fee
```

P0 实现建议：

- `pending_trading_fee` 可用订单冻结中的预估手续费上限；
- `pending_funding_fee` 可为当前批次待扣金额。

### 4.9 可交易余额

```text
available_balance =
equity
- Σ open_position_initial_margin_requirement
- Σ order_hold_requirement
- Σ withdraw_hold
- risk_buffer
```

说明：

- `risk_buffer` 为系统安全缓冲，可配置；
- 若 `available_balance < 0`，禁止风险增加订单。

### 4.10 风险率

```text
margin_ratio = equity / (Σ maintenance_margin_requirement + liquidation_fee_buffer)
```

其中：

```text
liquidation_fee_buffer = Σ (notional × liquidation_fee_rate)
```

#### 触发条件

```text
margin_ratio <= 1 -> 进入清算
```

### 4.11 资金费率

```text
funding_fee = position_notional_at_settlement × normalized_funding_rate
```

长仓支付：

```text
funding_fee > 0 -> LONG 支付, SHORT 收取
```

长仓收取：

```text
funding_fee < 0 -> LONG 收取, SHORT 支付
```

### 4.12 清算罚金

```text
liquidation_penalty = liquidated_notional × liquidation_penalty_rate
```

### 4.13 破产价值

账户级近似定义：

```text
bankruptcy_equity_threshold = 0
```

仓位展示级破产价应通过求解以下方程得到：

```text
equity_after_price_move = 0
```

P0 实施要求：

- 触发逻辑不依赖 `bankruptcy_price`；
- `bankruptcy_price` 仅作为前台展示与风险提示；
- 若展示无法精确求解，可采用数值法。

## 5. 清算逻辑规范

## 5.1 触发来源

- Mark Price 更新
- Funding 结算后
- 成交后仓位变化
- 管理员强制风险重算

## 5.2 清算前置动作

1. 将账户置为 `NO_NEW_RISK`
2. 撤销所有风险增加型挂单
3. 释放 `USER_ORDER_MARGIN`
4. 再次重算风险率

若释放挂单冻结后恢复安全，则退出清算流程并记录 `liquidation_aborted_after_release`。

## 5.3 P0 强平策略

P0 推荐：

- 默认先执行全平；
- 预留部分强平接口与状态字段；
- 若时间允许，可升级为“部分强平优先”。

原因：

- 题目时间有限；
- 先保证资金安全和状态一致；
- 再逐步优化用户体验与冲击成本。

## 5.4 强平成交价

强平成交也必须使用 `Execution Price`，但允许叠加更保守的 `liquidation_slippage_bps`。

```text
liquidation_execution_price =
normal_execution_price × (1 + extra_slippage)
```

方向说明：

- 平多卖出时用更差卖价；
- 平空买入时用更差买价。

## 5.5 清算顺序

1. 锁定账户资金与仓位快照
2. 撤单与解冻
3. 计算可平仓位
4. 生成 liquidation fill
5. 更新 position
6. 落账：损失、罚金、剩余返还
7. 若负权益，尝试保险基金兜底
8. 若保险基金不足，记录 `bad_debt` 并进入 ADL/人工处置
9. 发出审计与告警事件

## 5.6 保险基金使用条件

满足以下全部条件时允许动用保险基金：

- 清算结束后账户权益 < 0
- 用户 `USER_POSITION_MARGIN` 已耗尽
- 用户 `USER_WALLET` 已纳入跨账户权益计算
- 不允许再通过用户资金覆盖

## 5.7 ADL 边界

P0 要求：

- 数据模型与状态机必须预留；
- 后台与 Explorer 可查询；
- 自动执行可作为 P1。

## 6. Funding 批次规范

### 6.1 数据源

默认：

- Binance
- Hyperliquid

### 6.2 归一化

全部 funding 数据统一换算为 1 小时口径。

```text
normalized_rate = source_rate × source_interval_hours / 1h
```

### 6.3 聚合规则

```text
hourly_rate = clamp(avg(valid_normalized_rates), -funding_cap, funding_cap)
```

### 6.4 降级规则

- 双源有效：平均
- 单源有效：单源兜底
- 无有效源：symbol 切 `REDUCE_ONLY` 或暂停开新仓

### 6.5 幂等规则

批次唯一键：

```text
funding_batch_unique = symbol + time_window_start + time_window_end
```

明细唯一键：

```text
funding_item_unique = funding_batch_id + position_id
```

## 7. 并发与一致性要求

### 7.1 风险快照锁定

风控重算与强平执行不能并发修改同一账户资金。

推荐实现：

- 账户级分布式串行键：`risk_lock:{user_id}:{asset}`
- 数据库事务内锁定相关 `positions` 与 `account_balance_snapshots`

### 7.2 Funding 与交易并发

Funding 批次应用时，必须避免与同一仓位成交写入发生交叉覆盖。

推荐策略：

- 以 `position_id` 为粒度加锁
- 或者以 `user_id + symbol + side` 为串行键

### 7.3 强平与下单并发

账户进入 `NO_NEW_RISK` 后：

- 新开仓必须拒绝
- reduce-only 可允许
- 前端应收到状态变更推送

## 8. 审计与可追溯要求

每一次风险计算、清算、资金费率结算必须记录：

- 输入价格
- 输入仓位快照
- 输入账户快照
- 输入风险档位
- 公式版本
- 配置版本
- 输出值
- trace_id

## 9. 禁止事项

- 禁止以 Last Price 直接触发清算；
- 禁止以单一不健康行情源直接更新 Mark Price；
- 禁止 Funding 批次部分成功后标记成功；
- 禁止清算执行在没有快照与审计记录的情况下落账。
