# 实施准备度评估与补充清单

## 1. 结论

当前项目资料已经足以启动：

1. 系统级架构设计；
2. P0 范围界定；
3. 仓库骨架搭建；
4. 数据模型初稿；
5. API 分组与事件模型初稿；
6. 模块级开发排期。

但当前资料仍不足以直接支撑“完整大项目”的稳定实现与持续联调，原因不是缺少方向，而是仍缺少若干“不可模糊的实施级细节”。  
当前信息状态可定义为：

```text
适合开始 P0 架构落地与骨架实现
不适合直接无补充地进入完整生产级功能实现
```

## 2. 当前已具备的信息

### 2.1 已经明确的内容

- 题目目标、系统原则、账本约束；
- 主要业务模块；
- 链上托管、链下交易、链下风控、链下清算、外部对冲的总体边界；
- 技术栈约束；
- 顶层架构；
- 进程职责边界；
- 链上与链下状态归属；
- 数据流与核心状态机；
- 数据库实体主清单；
- API 和事件的主分组；
- P0 验收清单与测试矩阵；
- 里程碑级任务拆解。

### 2.2 这些信息已经足够支持的工作

- 初始化 monorepo 或多目录仓库结构；
- 初始化 Go 多进程工程骨架；
- 初始化 React 前端壳；
- 初始化 Vault 合约与 Foundry 工程；
- 初始化 MySQL/GORM 模型；
- 初始化 RabbitMQ 事件总线、Outbox、Redis 基础组件；
- 编写高层 Architecture Document；
- 建立模块边界与开发顺序。

## 3. 当前仍不足的信息

以下内容若不继续细化，后续会在编码过程中反复返工。

### 3.1 资金与账本层面的不足

1. 尚未形成“分录模板手册”。
2. 尚未固化每一种业务类型对应的借贷方向、子账户组合、幂等键规则。
3. 尚未明确补账、冲正、失败回滚在表结构与服务接口中的标准写法。
4. 尚未明确精度、舍入模式、最小单位、展示精度与内部精度之间的统一约束。

### 3.2 合约与链上托管层面的不足

1. Vault 合约接口、事件定义、权限模型、提现执行入口尚未细化为 ABI 级别。
2. 存款路径尚未明确是“每用户独立地址”还是“Vault + memo/映射地址”。
3. 充值事件监听范围尚未精确定义。
4. 合约升级策略、暂停策略、签名人/管理员角色边界尚未写成合约设计规范。

### 3.3 交易与价格层面的不足

1. Execution Price 公式还停留在原则层，没有落实到字段与算法输入输出。
2. 限价单、止损单、止盈单在 CFD 模式下的触发与成交规则仍需要逐字段确认。
3. symbol 精度、最小下单单位、交易时段、风险档位尚未形成初始配置清单。
4. 首期上线的 symbol 集合尚未冻结。

### 3.4 风控与清算层面的不足

1. 风险率、维持保证金、初始保证金、清算价格、破产价格需要形成统一公式文档。
2. 首期到底采用“直接全平”还是“部分强平优先”的实现决策需要冻结。
3. 负权益、保险基金、坏账、ADL 的边界需要明确为 P0、P1 还是预留。
4. 风险总闸和降级模式的触发规则与后台操作流程尚未形成 SOP。

### 3.5 对冲层面的不足

1. 对冲策略尚未从“模块职责”细化为“何时触发、怎么下单、如何幂等、如何补偿”。
2. Hyperliquid Testnet 账户、保证金镜像、连接器重试策略尚未冻结。
3. 对冲失败后的业务降级策略需要明确。

### 3.6 后端工程层面的不足

1. 模块内部 package 划分和依赖方向还未固化。
2. 同步 API 与异步 worker 的事务边界还未拆到 handler/service/repository 粒度。
3. 错误码、响应模型、鉴权中间件、审计埋点规范还未统一。
4. 配置中心的加载、缓存、热更新策略还未写成实现规范。

### 3.7 前端层面的不足

1. 页面信息架构、路由、状态管理边界还未明确。
2. 交易终端、账户中心、Explorer、后台各自的数据依赖尚未形成接口映射。
3. Loading、失败、重试、部分成功、异步处理中状态展示规范尚未定义。

### 3.8 运维与交付层面的不足

1. docker compose 中各服务最小启动集和依赖顺序尚未固化。
2. review 环境 mock 行情、faucet、模拟链事件的策略尚未落地。
3. 日志、指标、告警、死信、补偿与手工恢复流程尚未写成运维规范。

## 4. 现有文档是否还需要细化

需要。现有文档的定位仍以“系统设计”和“架构说明”为主，离真正开工所需的“实施规格”还差以下几层。

### 4.1 仍需补充的文档类型

1. 账本分录模板规范；
2. 合约接口与事件规格；
3. 精度与公式规范；
4. API 请求/响应 DTO 规范；
5. 事件 payload 规范；
6. Worker 重试、幂等、死信与恢复规范；
7. 配置项字典与默认值矩阵；
8. 初始 symbol 配置表；
9. 前端页面信息架构与接口映射表；
10. review 环境运行与演示剧本。

### 4.2 必须优先细化的文档

若以“立即开始开发”为目标，优先级最高的是以下六份：

1. 账本分录模板与资金状态机；
2. 风险与清算公式规范；
3. Vault 合约设计说明；
4. API/事件 Schema；
5. MySQL DDL 与索引设计；
6. 服务间事务与异步补偿规范。

## 5. 开始做之前还需要怎样拆解

开始编码前，应再做一层“实现级拆解”，把系统从模块级拆到“对象、流程、接口、表、事件、worker”的粒度。

### 5.1 推荐拆解层次

#### 第一层：业务对象冻结

- user
- account
- ledger_tx
- ledger_entry
- deposit
- withdrawal
- order
- fill
- position
- risk_snapshot
- liquidation
- funding_batch
- hedge_intent
- hedge_order
- hedge_fill
- audit_log
- config_item

#### 第二层：状态机冻结

逐个对象明确：

- 允许状态；
- 状态迁移触发条件；
- 持久化时机；
- 幂等键；
- 失败回滚路径；
- 审计事件。

#### 第三层：写链路冻结

逐条关键写链路明确：

- 哪些表在一个事务内提交；
- 哪些操作不能跨事务拆开；
- 哪些事件通过 outbox 异步发出；
- 哪些外部调用必须放在事务外；
- 外部调用成功但本地失败如何补偿。

#### 第四层：实现边界冻结

把每个模块下沉到：

- handler
- service
- repository
- worker
- connector
- scheduler
- model
- dto

#### 第五层：测试边界冻结

每条关键链路都需要明确：

- 单元测试；
- 事务测试；
- 并发测试；
- 回放测试；
- 异常恢复测试；
- E2E 测试。

## 6. 对“是否有充足信息”的判断

### 6.1 若目标是“开始做 P0 骨架”

信息是充足的。

可以立刻开始：

- 仓库初始化；
- 基础服务脚手架；
- MySQL schema 第一版；
- Vault 合约骨架；
- 登录/账本/订单主流程骨架；
- RabbitMQ Outbox 和 Worker 框架。

### 6.2 若目标是“完整做出生产级大项目”

信息还不充足。

仍需要补齐以下关键决策：

- 精确账本分录模板；
- 精确清算和资金费率公式；
- 合约 ABI 与事件；
- Hyperliquid 对冲策略；
- symbol 初始配置；
- 错误处理和补偿策略；
- review 环境演示路径与数据注入方案。

## 7. 推荐的补充顺序

建议按以下顺序继续细化，而不是同时铺开。

1. 先冻结账本、精度、状态机、公式。
2. 再冻结合约接口、MySQL DDL、API DTO、事件 Schema。
3. 再冻结每个 worker 的重试与补偿策略。
4. 最后补前端 IA、页面流和 review 演示流程。

当前最合理的推进方式不是直接全面编码，而是先完成以下实施级细化，再进入 Milestone 2 开发：

1. 账本分录模板；
2. 风险与清算公式；
3. Vault 合约设计；
4. API/事件 Schema；
5. DDL 初稿。

完成这一步后，项目将从“有架构设计”进入“可稳定开发”的状态。
