# 服务间事务、幂等与补偿规范

## 1. 文档目标

本文档定义 Backend API、Indexer、Trade Engine、Risk Engine、Liquidator、Hedger、Funding Worker 之间的通信、事务边界、幂等规则与补偿策略。

## 2. 总原则

1. 本地状态先于异步通知；
2. 外部调用必须在事务外；
3. 所有异步消费按至少一次投递设计；
4. 通过幂等键实现“效果上的 exactly-once”；
5. 任何跨边界失败都必须有补偿或人工恢复路径。

## 3. 强一致写链路

必须单事务提交：

- 登录会话创建
- 提现冻结
- 内部转账
- 订单接受 + 冻结
- fill + position + ledger
- 清算执行
- funding item 应用

## 4. Outbox 规范

### 4.1 事务内写出

每次关键事务必须写：

- 业务表
- 账本表
- `outbox_events`

### 4.2 Relay

`outbox-relay` 轮询 `outbox_events`，发布至 RabbitMQ。

### 4.3 幂等发布

- `event_id` 全局唯一
- relay 可重复投递同一 `event_id`
- consumer 必须幂等

## 5. Consumer 幂等规范

### 5.1 记录表

每个 consumer 必须有消费去重记录，建议表：

- `message_consumptions`

唯一键：

```text
consumer_name + event_id
```

### 5.2 处理流程

1. 收到消息
2. 开事务
3. 插入 `message_consumptions`
4. 若唯一键冲突，则 ack 并返回
5. 执行业务逻辑
6. 提交事务
7. ack 消息

## 6. 重试策略

### 6.1 可自动重试

- MQ 临时失败
- Hyperliquid 临时超时
- RPC 节点短暂不可用
- 行情源瞬时失败

### 6.2 不可自动重试

- 幂等冲突且请求体不一致
- 风险校验失败
- 合约权限错误
- 配置缺失
- 数据约束错误

### 6.3 Backoff

建议：

- 第 1 次：立即
- 第 2 次：5s
- 第 3 次：30s
- 第 4 次：2m
- 第 5 次：10m

超过阈值进入死信队列。

## 7. 死信队列

每个关键队列必须有 DLQ：

- `ledger.dlq`
- `risk.dlq`
- `liquidation.dlq`
- `funding.dlq`
- `hedge.dlq`
- `indexer.dlq`

死信消息必须带：

- 原始 `event_id`
- 失败原因
- 重试次数
- 最后处理节点
- trace_id

## 8. 关键补偿场景

### 8.1 广播成功，本地失败

场景：

- 提现链上广播成功；
- 数据库事务失败。

策略：

- 记录孤儿广播告警；
- Indexer 监听链上事件进行回补；
- 回补前检查 `withdraw_id` 是否已完成；
- 回补过程使用 `withdraw_broadcast:{withdraw_id}:{tx_hash}` 幂等键。

### 8.2 MQ 已发布，消费者处理失败

策略：

- 依赖 consumer 幂等重试；
- 超阈值进入 DLQ；
- 人工重放时按 `event_id` 幂等。

### 8.3 对冲成功，本地失败

策略：

- 通过 Hyperliquid venue order/fill 回查重建本地状态；
- 以 `venue_fill_id` 为镜像账务幂等键；
- 风险层在未确认对冲状态前可启用更保守阈值。

### 8.4 清算触发重复投递

策略：

- `liquidation_id` 唯一；
- `liquidation_items` 以 `(liquidation_id, position_id)` 去重；
- 已执行的清算再次触发只返回既有结果。

## 9. 通信方式矩阵

### 9.1 同步 API

- 前端 -> API
- API -> Auth/Order/Ledger/Account service
- 管理后台 -> Config/Admin service

### 9.2 异步任务

- Indexer -> deposit events
- Trade Engine -> fill/position events
- Risk Engine -> liquidation/hedge request events
- Funding Worker -> funding batch events
- Hedger -> hedge update events
- Relay -> notification/explorer read models

## 10. RabbitMQ 拓扑建议

Exchange：

- `domain.events`
- `ops.events`

Routing keys：

- `wallet.deposit.*`
- `wallet.withdraw.*`
- `trade.order.*`
- `trade.fill.*`
- `risk.liquidation.*`
- `risk.funding.*`
- `hedge.*`
- `config.*`
- `audit.*`

## 11. 事务外调用清单

以下调用必须在数据库事务外：

- 链上签名与广播
- Hyperliquid 下单
- 外部行情 HTTP 回补
- 通知发送

## 12. 审计要求

每次跨边界调用必须记录：

- request id
- trace_id
- upstream/downstream
- payload hash
- retry count
- result
- duration

## 13. 禁止事项

- 禁止在数据库事务内调用外部 HTTP/RPC；
- 禁止依赖 MQ 保证 exactly-once；
- 禁止没有幂等键的消费者落账；
- 禁止把补偿逻辑留给人工而没有系统告警与回补入口。
